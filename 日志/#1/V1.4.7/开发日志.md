## 关键发现

- `gitnexus@1.6.3` 在本机与极小测试 repo 上会崩溃，退出码 `-1073741819`。
- GitHub issue #1406 中维护者建议遇到 `1.6.3` embeddings 失败时先使用 rc。
- `gitnexus@1.6.4-rc.84` 可完成 CarbonRag 全量索引。
- 使用 Clash 代理 `http://127.0.0.1:17891` 后，下载 HuggingFace 模型和 LadybugDB 扩展更稳定。
- 如果不设置代理或镜像，`--embeddings` 可能报 `fetch failed`，失败位置通常在 HuggingFace transformers 模型下载。
- Windows 当前会提示 `VECTOR index` 不支持，并退回 exact-scan fallback；这不是失败，语义查询仍可用，只是性能降级。

## #2/#3 可复刻步骤

### 1. 安装 GitNexus rc

不要使用 `gitnexus@latest`，因为当前 latest 是 `1.6.3`，本机已复现崩溃。

```powershell
npm install -g gitnexus@rc
gitnexus --version
```

预期版本：

```text
1.6.4-rc.84
```

### 2. 注册 Codex MCP

```powershell
codex mcp remove gitnexus
codex mcp add gitnexus -- npx -y gitnexus@rc mcp
codex mcp list
```

预期能看到：

```text
gitnexus    npx      -y gitnexus@rc mcp
```

### 3. 使用 Clash 代理跑完整索引

#1 本机 Clash 端口为 `17891`。如果 #2/#3 的 Clash 端口不同，改成自己的端口。

```powershell
powershell -NoProfile -ExecutionPolicy Bypass -File scripts/gitnexus-full-index.ps1 -Proxy "http://127.0.0.1:17891" -HfEndpoint "https://huggingface.co"
```

如果没有代理，先尝试默认镜像：

```powershell
powershell -NoProfile -ExecutionPolicy Bypass -File scripts/gitnexus-full-index.ps1
```

### 4. 验收命令

```powershell
gitnexus status
gitnexus list
gitnexus query carbon --limit 3
gitnexus impact CarbonCalculationEngine
gitnexus detect_changes
```

### 5. Web UI 验证

```powershell
gitnexus serve --host 127.0.0.1 --port 4747
```

浏览器打开：

```text
http://127.0.0.1:4747
```

如果用脚本后台启动，记得关闭对应 `node.exe`，避免占用 `4747`。

#1 二次复测结果：

- 重跑索引后当前提交为 `e35dfe9`，状态 `up-to-date`。
- 最新索引：`8,230 nodes / 16,087 edges / 339 clusters / 300 flows`。
- `/api/repos` 返回 CarbonRag 仓库元数据。

## 本轮踩坑与处理

| 问题 | 现象 | 处理 |
| --- | --- | --- |
| `gitnexus@1.6.3` 崩溃 | 极小 repo 也返回 `-1073741819` | 改用 `npm install -g gitnexus@rc` |
| embeddings 下载失败 | `Analysis failed: fetch failed` | 使用 `HF_ENDPOINT=https://hf-mirror.com` 或 Clash 代理 |
| LadybugDB FTS/VECTOR 下载/加载问题 | FTS unavailable 或 VECTOR exact-scan fallback | 代理可解决 FTS 下载；VECTOR fallback 暂接受 |
| PowerShell `Start-Process gitnexus` 失败 | “不是有效的 Win32 应用程序” | 用 `gitnexus.cmd` 或直接前台运行 `gitnexus serve` |
| `.gitnexus/` 出现半成品 WAL | database id mismatch | 删除 `.gitnexus/` 后重跑索引 |
| `detect_changes` 输出 `.gitnexus/lbug` 文件锁 | Web UI 的 `gitnexus serve` 仍在运行 | 停掉 4747 对应 node 进程后重跑 |
