# V1.3.2 #2 开发日志

## 本轮目标

实现 `policy-knowledge-three-stage-ingestion` 的第一层政策采集能力：在默认关闭和可选依赖前提下，补齐 Scrapy 最小真实爬虫 runner，并验证本地爬虫功能可用。

## 实际完成内容

- 新增政策三段式摄取 OpenSpec change：`policy-knowledge-three-stage-ingestion`。
- 新增官方政策采集边界：`PolicyCrawlRequest`、`CrawledDocument`、`CrawlerProvider`、`FakeCrawlerProvider`、`ScrapyCrawlerProvider`。
- `ScrapyCrawlerProvider` 在显式启用且安装 Scrapy 时，可通过独立子进程 runner 真实抓取 allowlist 页面。
- 新增本地 HTTP fixture 烟测，验证 Scrapy 能抓取入口页并跟随同域链接。
- 保持 `RAG_ENABLE_POLICY_CRAWLER=false` 默认关闭，Scrapy 仅记录在 `backend/requirements-optional.txt`，不作为强制依赖。
- 新增 HTML/PDF/OFD 解析路由、政策治理 metadata、`public_policy_web` 条目类型、chunk metadata 存储和 `policy_ingest` workflow 边界。
- 补齐采集结果入库闭环：`CrawledDocument` 可进入 `KnowledgeService.create_policy_item_from_crawled_document`，创建或刷新共享 `public_policy_web` 知识条目并入队 `crawl_ingest`。
- `crawl_ingest` 现在通过 `policy_ingest` workflow 完成 staging 校验、HTML/PDF/OFD 解析路由、政策治理 metadata、chunk 构建和 runtime chunk 索引。
- 公共政策 BM25 检索会追加已索引的 `public_policy_web` runtime chunks，对外仍保持 `public_policy` evidence 形态。
- 完成 Slice G 解析路由硬化：
  - HTML 正文提取会跳过 `nav`、`footer`、`sidebar`、`share`、`breadcrumb`、`script`、`style` 等常见噪声。
  - HTML 轻量提取 `publication_date`、`source_label`、`issuing_authority`，并进入 `ParsedDocument.metadata`。
  - PDF 通过现有 `ParserRegistry` 解析后统一包裹为 `source_type=public_policy_web`，并保留 `parser_chain`、`source_url`、`original_source_uri`。
  - OFD 仍为可选转换，转换后的 PDF/HTML 继续走 parser registry，同时保留 `converted_from`、`converter_name`、`converter_available` 等 metadata。
  - `crawl_ingest` 生成的 chunk metadata 中保留 parser 信息和 parser chain。
- 完成 OpenSpec archive：`policy-knowledge-three-stage-ingestion` 已归档到 `openspec/changes/archive/2026-05-07-policy-knowledge-three-stage-ingestion/`，delta spec 已合入 `openspec/specs/knowledge-rag/spec.md`。
- 新增稳定验证脚本 `scripts/verify_policy_ingestion.py`，避免在 Windows PowerShell here-string / 管道执行时中文 fixture 被终端编码破坏，导致发布日期、关键词检索和 BM25 命中误判失败。

## 已知限制

- 本轮只验证本地 fixture 爬取，不默认启用公网抓取。
- Scrapyd 调度、真实 OFDRW 转换、后台政策源管理 UI 仍在后续切片中。
- CI 若未安装 Scrapy，真实 Scrapy 烟测会按 optional dependency 设计跳过。
- 目前还没有后台 UI 直接触发政策采集入库；本轮提供的是服务层入口、任务闭环和可检索验证。
- HTML 清洗规则保持保守启发式，不替代后续更强的正文抽取器；如果官方站点模板差异较大，需要用 fixture 继续扩充规则。

## Showcase Readiness 补充

- 新增 OpenSpec change：`policy-ingestion-showcase-readiness`，目标从“单独 demo”调整为“现有产品面达到可演示级别”。
- 新增内置离线合成演示源 `low-carbon-campus-action`，包含演示 HTML、内部 source URI、演示编号、发布日期、施行日期和低碳/碳核算/节能改造等政策术语；该源不是官方政策文件。
- 新增管理员 API：
  - `GET /api/v1/admin/policy-sources`
  - `POST /api/v1/admin/policy-sources/{source_id}/run`
  - `GET /api/v1/admin/policy-sources/{source_id}/status`
  - `GET /api/v1/admin/policy-sources/{source_id}/chunks`
  - `GET /api/v1/admin/policy-sources/{source_id}/retrieval-preview`
- 管理员控制台已集成“政策知识三段式摄取”区域，可直接运行/刷新内置政策源，查看 source、task、workflow、metadata、chunks 和 retrieval preview。
- 重复运行会刷新同一个 `public_policy_web` 条目，不会无限创建重复知识条目。
- 检索预览通过公共政策检索链路返回 `public_policy_demo` evidence shape，保持 `/ask`、RAG Lab、retrieval-only、calc/report/session 默认行为隔离，且不伪装成官方 `public_policy` 引用。

## Showcase 验证步骤

1. 启动后端和前端后，以管理员账号进入 `/admin`。
2. 在“政策知识三段式摄取”区域点击“运行/刷新政策摄取”。
3. 确认状态变为“已入库可检索”，最新任务为 `crawl_ingest` 且 succeeded。
4. 确认 workflow 为 `policy_ingest / completed`，节点包含 `crawl_source`、`stage_crawled_document`、`parse_document`、`normalize_policy_metadata`、`build_chunks`、`upsert_vector_index`、`index_completed`。
5. 确认政策分块显示 `public_policy_demo`，并带有演示来源、发布日期、样例编号、条款锚点等 metadata。
6. 确认检索预览问题“低碳韧性校园 碳核算 节能改造”能命中“本次政策源”。
7. 可打开 RAG Lab，用同一问题进行公共政策检索验证。
8. 本地脚本验证：`cd backend && .\.venv\Scripts\python.exe ..\scripts\verify_policy_showcase.py`。
