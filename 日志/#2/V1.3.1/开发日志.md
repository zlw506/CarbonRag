# V1.3.X #2 RAG initial baseline 开发日志

## 本轮目标

按最新 `Git-ys1/CarbonRag` 开发公告要求，将 #2 的 V1.3.x RAG / LightRAG-inspired 主线整理为阶段性 PR，并补齐发布前开发日志。本 PR 不是单一 V1.3.1 patch，而是 V1.3.X RAG initial baseline。

## 实际完成内容

- 建立 CarbonRag-native RAG engine 骨架和 RAG Lab，用于 retrieval-only 验证。
- 增强 RAG Lab 可观测性：后端地址、请求参数、检索模式、0 命中、fallback metadata、vector/graph/rerank 状态。
- 新增统一 RAG 数据契约：`ParsedDocument`、`DocumentBlock`、`ChunkRecord`、`EmbeddingRecord`、`CitationRef`、`RetrievalTrace`。
- 新增 ParserProvider、VectorStoreAdapter、HybridRetriever、RerankProvider、GraphIndexBuilder、graph query modes 等实验边界。
- 新增可选 Docling / MinerU / pgvector adapter 预留，默认关闭，不作为本地强依赖。
- 新增 RAG 检索评测集和 `scripts/rag_eval.py` baseline。
- 新增轻量 RAG ingest workflow 与治理字段预留，记录上传、解析、切块、索引、图谱候选等节点状态。
- 按 #1 review request 新增 `rag-retrieve-isolation-hardening`：private/mixed 空选择 fail-closed、retrieval-only 只读、安全错误响应、RAG Lab 不展示后端内部异常。
- 合入 upstream/main 最新 V1.4.5 开发公告和碳核算可信底座，按公告补充本开发日志。

## 已知限制

- Docling、MinerU、pgvector、Graph query modes 均为可选或实验能力，默认不改变 `/ask` 行为。
- GraphRAG、Neo4j、LangGraph、Keycloak/OpenFGA、K8s 等企业级重依赖未接入。
- retrieval-only 不再在请求路径同步执行 ingest/rebuild；新上传知识必须等待知识任务完成索引后再被检索。
