## 一句话结论

企业级 RAG 知识库不是“上传文档 + 调模型”，而是一套围绕数据治理、文档解析、切片策略、向量索引、权限隔离、检索重排、提示词生成、效果评测和持续运营的工程系统。真正的难点不在把流程跑通，而在让答案准确、可控、可追溯、可扩展、可交付。

## 适合做 RAG 的场景

优先选择这些场景：

- 企业制度、产品手册、客服知识库、售后文档、技术文档、合同模板、项目资料等知识密集型问答。
- 问题答案依赖企业私有资料，通用大模型无法直接回答或容易幻觉。
- 文档更新频繁，需要比模型训练更轻量的知识注入方式。
- 用户需要答案引用来源，便于人工核验。
- 业务流程有明确输入、输出、规则和可评测标准。

谨慎选择这些场景：

- 数据质量差、缺少权威来源，连人工也难以判断标准答案。
- 问题主要依赖复杂推理、实时业务状态或多系统事务操作，单纯 RAG 不足以解决。
- 企业无法提供权限体系、文档归属、密级和访问控制规则。
- 只追求“能聊天”，没有明确业务指标。

企业场景评估可以从五个维度打分：

| 维度 | 判断问题 |
| --- | --- |
| 可结构化程度 | 是否有明确规则、流程、模板、知识边界 |
| 重复性 | 是否有大量重复问答、查询、判断任务 |
| 数据可得性 | 是否能拿到历史文档、案例、标准答案和业务数据 |
| 人机交互性 | 用户是否会频繁查询、追问、反馈 |
| AI 带来的价值 | 是否能提升效率、降低成本、减少人工错误 |

## 企业级 RAG 总体架构

典型企业级 RAG 可以拆成两条链路：数据入库链路和用户问答链路。

### 数据入库链路

1. 数据接入：PDF、Word、Markdown、TXT、HTML、网页、图片、表格、数据库记录、知识平台文档。
2. 文档解析：抽取正文、标题、段落、表格、图片 OCR、页码、章节层级。
3. 清洗标准化：去广告、去页眉页脚、去重复内容、修正编码和无效字符。
4. 切片 chunking：按语义、标题、段落、长度、重叠窗口切分。
5. 元数据绑定：文档 ID、来源、作者、部门、密级、权限标签、时间、版本、页码。
6. embedding 向量化：将文本片段转换为向量。
7. 索引入库：写入向量数据库，同时保存原文、摘要、元数据和权限信息。

### 用户问答链路

1. 用户提问：接收问题、用户身份、部门、角色、会话上下文。
2. 查询改写：必要时做意图识别、问题补全、同义词扩展、多轮上下文改写。
3. 权限解析：从权限中心获取用户可访问的部门、角色、文档密级、项目范围。
4. 向量召回：将问题向量化，在向量库中按相似度召回候选片段。
5. 权限过滤：通过向量库 metadata/filter 能力下推过滤。
6. 混合检索：结合向量检索、关键词检索、结构化条件过滤。
7. 重排 rerank：用 CrossEncoder 或 rerank 模型重新排序候选片段。
8. 上下文组装：控制 token 长度，合并相邻片段，保留来源引用。
9. 生成答案：将问题、上下文、规则提示词交给大模型。
10. 答案后处理：补充引用、置信度、未命中提示、敏感信息检查。
11. 记录反馈：保存问题、召回片段、答案、用户反馈，用于评测和优化。

## 核心模块设计

### 1. 文档接入

企业知识库的质量首先取决于数据接入。常见数据源包括：

- 本地文件：PDF、Word、Excel、PPT、Markdown、TXT。
- 企业平台：Confluence、飞书、钉钉、语雀、SharePoint、Notion。
- 业务系统：CRM、ERP、工单系统、产品系统、内部数据库。
- 多媒体资料：截图、扫描件、图片、视频字幕、会议纪要。

接入时要记录文档来源、更新时间、版本号和责任人，否则后续很难追踪错误答案来自哪里。

### 2. 文档解析

不要把解析看成简单读取文本。企业文档经常包含表格、图片、页眉页脚、目录、编号、跨页内容和复杂层级。

有用做法：

- PDF 简单文本可以用 PyPDF、pdfplumber 等解析。
- 扫描件或图片 PDF 需要 OCR。
- 表格应尽量保留行列关系，不要直接压成混乱文本。
- 标题、章节、页码、表格说明应作为 metadata 保存。
- 对合同、制度、手册这类文档，要保留条款编号。

需要避免：

- 只按纯文本抽取，丢掉表格结构。
- 把页眉、页脚、版权声明、目录重复入库。
- 图片中的关键文字没有 OCR。
- 文档版本不清，旧制度和新制度同时命中。

### 3. 文本切片

切片是 RAG 效果的核心变量之一。切得太小，语义不完整；切得太大，召回噪声多且占用 token。

常用策略：

- 固定长度切片：实现简单，适合快速验证，但容易切断语义。
- 按标题和段落切片：更符合文档结构，适合制度、手册、教程。
- 滑动窗口切片：通过 overlap 保留上下文，减少信息断裂。
- 父子切片：小片段用于召回，大片段用于生成上下文。
- 表格专用切片：按表名、字段、行列语义组织，保留表头。

建议起步参数：

- 中文知识库 chunk size 可从 300-800 字试起。
- overlap 可从 50-150 字试起。
- 对制度条款、FAQ、API 文档，优先按自然结构切。
- 对长教程和白皮书，可以先按章节，再按段落切。

### 4. Embedding 模型

embedding 模型负责把文本转换成向量。它决定“语义相似”的基础质量。

选择时关注：

- 中文语义效果。
- 向量维度和存储成本。
- 是否支持长文本。
- 是否支持本地部署。
- 与向量数据库和业务框架的兼容性。

注意事项：

- 文本和图片通常需要不同 embedding 模型。
- 不同 embedding 模型生成的向量不能混用在同一 collection 中。
- 模型更换后，已有文档通常需要重新向量化和重建索引。
- 对企业资料，优先用评测集验证 embedding 效果，不要只看排行榜。

### 5. 向量数据库

向量数据库用于存储向量并执行相似度检索。选型时不要只看能不能存向量，还要看 metadata 过滤、索引性能、扩展性和运维复杂度。

重点能力：

- 向量相似度检索。
- metadata 标量过滤。
- collection/namespace 隔离。
- 批量导入和增量更新。
- 删除、重建索引、版本管理。
- 高并发查询性能。
- 本地部署或云服务能力。

设计建议：

- 不同业务域、不同 embedding 模型、不同模态可以拆 collection。
- 文档原文、向量、metadata、权限标签要能关联回同一个 chunk ID。
- 大规模数据不要在业务层做大量循环过滤，应尽量下推到向量库或检索引擎。

### 6. 召回策略

单纯向量召回不一定够。企业场景常需要混合检索。

常见召回方式：

- 向量召回：适合语义相似问题。
- 关键词召回：适合专有名词、编号、条款、错误码、产品型号。
- 结构化过滤：适合部门、项目、时间、文档类型、密级。
- 多路召回：多种召回结果合并去重，再统一重排。
- 查询扩展：补充同义词、缩写、别名。

实践建议：

- topK 不宜过小，否则容易漏召回。
- 召回阶段可以多拿一些候选，再交给 rerank 精排。
- 命中特定编号、制度条款、API 名称时，关键词检索往往比向量检索更可靠。
- 多轮问答要把上下文改写成独立问题再检索。

### 7. 重排 rerank

召回模型负责“先找一批可能相关的”，重排模型负责“从候选里挑更相关的”。

常见流程：

1. 向量库召回 top20 或 top50。
2. rerank 模型对“问题 + 候选片段”逐对打分。
3. 选择 top3-top8 进入大模型上下文。

适合引入 rerank 的情况：

- 知识库文档多、相似内容多。
- 向量召回命中但排序不稳定。
- 答案经常引用了不够相关的片段。
- 用户问题比较短，语义不充分。

### 8. Prompt 和答案生成

生成阶段不要只把片段拼给大模型。企业问答需要明确约束。

提示词应包含：

- 角色：你是企业知识库助手。
- 回答依据：只基于给定上下文回答。
- 不知道策略：上下文不足时说明无法从知识库确认。
- 引用要求：列出来源文档、章节、页码或 chunk 来源。
- 安全要求：不得输出无权限内容、敏感信息或猜测结论。
- 格式要求：按步骤、表格、结论、依据输出。

上下文组装建议：

- 优先保留高分片段。
- 合并相邻 chunk，避免答案缺上下文。
- 去除重复片段。
- 控制总 token，避免上下文过长稀释重点。
- 保留引用 ID，方便答案追溯。

## 权限隔离设计

权限隔离是企业级 RAG 和个人知识库最大的差异之一。

### 错误方案一：先检索后过滤

流程：

1. 用户提问。
2. 先从向量库检索 top10。
3. 再到业务系统判断这 10 条有没有权限。
4. 无权限的剔除，剩下的交给大模型。

问题：

- 如果 top10 都是高密级文档，过滤后可能一条不剩。
- 明明知识库里有用户可见的低密级相关文档，却被无权限文档挤出 topK。
- 这会造成“漏斗坍塌”，召回质量很差。

### 错误方案二：先查全量权限 ID 再检索

流程：

1. 先从关系库查出用户有权限的所有文档 ID。
2. 带着几万甚至几十万个 ID 去向量库检索。

问题：

- 查询条件过长。
- 并发压力大。
- 向量检索性能明显下降。
- 权限集合越大，系统越容易成为性能瓶颈。

### 推荐方案：权限元数据下推

核心思路：文档入库时就把权限标签写入 metadata，检索时把用户权限条件传给向量库，让过滤在向量检索内部完成。

入库时绑定：

- tenant_id：租户 ID。
- department_id：部门 ID。
- project_id：项目 ID。
- role_ids：可访问角色。
- user_ids：指定可访问用户。
- security_level：密级。
- doc_status：生效、废弃、草稿。
- valid_from / valid_to：制度有效期。

查询时执行：

1. 用户登录后获得 user_id、role_ids、department_id、tenant_id。
2. 权限中心返回该用户可访问的权限标签。
3. 向量检索时带上 metadata filter。
4. 向量库在合法数据范围内计算相似度和 topK。
5. 返回的片段天然满足权限约束。

一句话原则：不要在业务层对大批量候选做权限循环过滤，要把权限条件作为 metadata 下推到检索引擎。

## 本地化与工具方案

### RAGFlow 方案

RAGFlow 适合快速搭建可视化知识库，用于验证文档解析、切片、embedding、检索和问答流程。

适合：

- 个人知识库。
- 企业 PoC。
- 非技术人员参与文档上传和测试。
- 快速比较不同文档解析策略。

注意：

- PoC 跑通不等于生产可用。
- 生产环境仍要补齐权限、审计、评测、备份、监控、灰度发布。
- 如果涉及企业敏感资料，要确认部署方式、模型调用方式和数据存储位置。

### Python 手写方案

用 Python 手写一遍 RAG 能帮助理解每个模块的输入输出。

最小闭环：

1. 读取文档。
2. 文本切片。
3. 调 embedding 模型生成向量。
4. 存入向量库。
5. 用户问题向量化。
6. 相似度检索。
7. rerank。
8. 拼接 prompt。
9. 调用大模型生成答案。

适合：

- 学习底层流程。
- 写定制化接口。
- 接入企业现有系统。
- 做自动化评测和批处理。

## 企业落地关键工程问题

### 数据更新

企业文档不是一次性导入就结束。需要支持：

- 增量同步。
- 文档删除和失效。
- 版本替换。
- 重新切片和重新 embedding。
- 索引重建。
- 数据回滚。

### 答案可追溯

每个答案最好能显示：

- 来源文档标题。
- 来源章节或页码。
- chunk ID。
- 相似度或 rerank 分数。
- 文档更新时间。

这样用户可以判断答案是否可信，管理员也能定位错误来源。

### 敏感信息控制

除了检索权限，还要做生成阶段的安全控制：

- 检查上下文是否越权。
- 检查答案是否包含敏感字段。
- 对涉密内容做脱敏。
- 对无法确认的问题拒答或转人工。
- 记录审计日志。

### 多租户隔离

企业级系统通常要支持多租户或多部门隔离：

- 租户级 namespace 或 collection。
- metadata 中保存 tenant_id。
- API 层校验 tenant_id。
- 日志和评测数据按租户隔离。
- 备份恢复不能跨租户污染。

### 可观测性

至少记录：

- 用户问题。
- 改写后的查询。
- 召回片段。
- rerank 分数。
- 最终上下文。
- 模型输入输出。
- 耗时。
- token 用量。
- 用户反馈。

没有这些日志，就很难优化“为什么答错”。

## 评测指标

企业级 RAG 必须评测，不然只能靠感觉。

### 检索指标

- Recall@K：标准答案相关片段是否进入 topK。
- Precision@K：topK 中有多少是真相关。
- MRR：第一个相关结果排在多靠前。
- 命中率：问题是否能召回任何有效片段。

### 生成指标

- 答案准确率。
- 引用正确率。
- 幻觉率。
- 拒答正确率。
- 格式合规率。
- 用户满意度。

### 工程指标

- 平均响应时间。
- P95 / P99 延迟。
- 并发能力。
- 向量库查询耗时。
- embedding 成本。
- 大模型 token 成本。
- 索引构建耗时。

## 常见问题和处理办法

| 问题 | 常见原因 | 优化方向 |
| --- | --- | --- |
| 答案胡编 | 上下文不足、prompt 未约束、召回错误 | 加强拒答策略、优化召回和 rerank |
| 找不到明明存在的内容 | 切片不合理、embedding 效果差、topK 太小 | 调整切片、引入关键词检索、增加 topK |
| 引用不准确 | chunk 来源信息丢失、上下文拼接混乱 | 保存页码章节，生成时强制引用来源 |
| 权限越权风险 | 业务层后过滤、metadata 不完整 | 权限元数据入库，检索时下推过滤 |
| 响应慢 | 文档过多、召回过宽、rerank 成本高 | 多级索引、缓存、限制候选数、异步处理 |
| 答案太碎 | chunk 太小、没有合并相邻片段 | 使用父子切片或上下文扩展 |
| 答案过泛 | 文档质量差、问题太宽泛 | 查询改写、追问澄清、限定知识范围 |

## 推荐开发路线

### 第一阶段：PoC 验证

- 选择一个边界清晰的业务场景。
- 准备 50-200 个高质量文档。
- 整理 30-100 个真实问题和标准答案。
- 用 RAGFlow 或 Python 快速跑通闭环。
- 比较不同切片、embedding、topK、rerank 配置。

交付目标：证明 RAG 对该场景有价值。

### 第二阶段：MVP 系统

- 接入真实用户登录和权限系统。
- 建立文档上传、解析、入库流程。
- 支持基础问答和来源引用。
- 增加日志、反馈、错误追踪。
- 建立最小评测集。

交付目标：让小范围业务用户可用，并能持续收集反馈。

### 第三阶段：生产化

- 完成权限隔离、多租户、审计。
- 支持增量同步、版本管理、删除失效。
- 建立自动评测和灰度发布流程。
- 优化性能、成本和稳定性。
- 接入监控告警和运营后台。

交付目标：系统可稳定服务真实业务。

### 第四阶段：持续优化

- 根据用户反馈扩充评测集。
- 分析 bad case，定位是解析、切片、召回、重排还是生成问题。
- 对高频问题建立 FAQ 或缓存。
- 针对业务域优化词表、同义词、规则和 prompt。
- 定期重建索引，清理废弃文档。

交付目标：让知识库随着业务变化持续变好。
