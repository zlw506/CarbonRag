  - 当前 provider 已有 SSE 聚合底子，但 ask 仍是“后端整段返回、前端整段落地”
  - 本地学习源固定为 `F:\Project\CarbonRag\CarbonRag\3rdparty\claw-code-study\fix_ui_parity_snapshot`
  - 云端参考固定为 `https://github.com/ultraworkers/claw-code/tree/fix/ui-parity`

  - 前端流式消费：原生 `fetch + ReadableStream + SSE parser`，不再引入额外前端流式依赖
  - thinking 文本只作为运行中 UI 数据，不持久化进 session 正文

## Key Changes
- 主线与学习收窄：
  - housekeeping 只做“确认并文档化”当前 `main` / `release/cloud-stable` 已对齐到 `v1.1.3`，不再把这一步当本轮主工作。
  - 新增 `docs/research/claw-code/07_CHAT_UX_AND_MEMORY_NOTES.md`，只沉淀 3 件事：
    - `provider abstraction + streaming` 如何映射到 CarbonRag
    - `session state / compaction` 如何映射到当前 ask
    - `memory / skills / hooks` 与 shared/local config 边界，哪些借了、哪些没借
  - 继续把 shared defaults / local overrides 写进文档口径：
    - 共享配置继续落仓库模板和稳定分支
    - 本机覆盖继续留本地 env / 本地脚本，不把机器状态混进共享主线

- 后端流式 ask foundation：
  - 保留现有 `POST /api/v1/sessions/{id}/ask` 非流式接口。
  - 新增 `POST /api/v1/sessions/{id}/ask/stream`，返回 `text/event-stream`。
  - provider 层升级为显式流式抽象，不再只暴露“聚合后的完整字符串”：
    - 新增统一 `ChatStreamEvent`
    - 事件类型固定为：`status`、`thinking_delta`、`answer_delta`、`done`、`error`
    - `generate_response()` 改为基于 `stream_response()` 聚合，避免双套逻辑
  - `OpenAICompatibleChatProvider` 直接解析上游 SSE：
    - 普通内容片段映射为 `answer_delta`
    - reasoning / thinking 相关字段映射为 `thinking_delta`
    - 若上游没有 thinking 片段，也必须先发 `status=thinking`
    - provider 只负责 chunk 分类，不决定 UI 呈现
  - `AIRuntimeOrchestrator` 增加流式运行入口：
    - 保留 `run()` 兼容非流式
    - 新增 `run_stream()`，负责 tool 执行、context build、provider stream 分发、最终 citations/source_summary 收尾
    - `run_stream()` 必须在开始回答前完成 grounding 和 session compaction
  - `sessions/{id}/ask/stream` 路由行为固定为：
    - 先校验 session / auth / scope /长度
    - 立即持久化用户消息
    - 立即创建一条空内容 assistant 消息，占位状态为 `pending`
    - 开始 SSE 输出
    - 流结束时更新同一条 assistant 消息为最终正文、citations、source_summary、trace_id、状态
    - 若 provider 失败，则把该 assistant 消息更新为 `provider_error` 和受控错误文案
  - SSE 事件线固定为：
    - `message_start`：返回 `user_message_id`、`assistant_message_id`、`trace_id`
    - `status`：`pending | thinking | streaming | done | error`
    - `thinking_delta`
    - `answer_delta`
    - `metadata`：最终 `citations`、`source_summary`、`memory_state`
    - `done`
    - `error`
  - 思考文本默认不落库；重开 session 只保留最终 assistant 消息和 citations，不回放 thinking 过程。

- Session memory 体感增强：
  - 保留 V1.1.3 的 `session_summary + recent_messages + memory_notes` 基础，不另起新 memory 模块。
  - 本轮把 context build 策略写死并可观测：
    - 总是保留当前问题
    - 优先保留 `session_summary`
    - 再保留最近 6 轮完整 `user/assistant`
    - 再拼 grounding 结果
    - `memory_notes` 最后读入，且超预算时最先裁剪
  - 超预算裁剪策略固定为：
    - 先截断/减少 `memory_notes`
    - 再减少 grounding 条数
    - 最后才从最近窗口最旧轮次开始裁
    - 不裁 `session_summary` 和当前问题
  - 自动 compaction 触发仍在 ask 前执行，但本轮前端要把它显性化：
    - 当前回答使用了“最近 N 轮 + session summary + M 条依据”
    - 如果本轮触发了 compaction，消息区显示“已压缩旧对话，保留最近 6 轮完整消息”
  - `GET /api/v1/sessions/{id}` 的 `memory_state` 保留并继续增强展示，不新增 memory 管理页。

- 前端聊天体验重做：
  - Ask 页仍是当前工作台，不新造页面。
  - 发送行为固定为：
    - 用户点击发送后，用户消息立即入流
    - 输入框立即清空
    - assistant 占位消息立即插入
    - 该占位消息按状态机流转：`pending -> thinking -> streaming -> done/error`
  - 前端不等待整段返回后再刷新整个 session，而是本地维护当前 streaming message：
    - `pending`：刚创建占位
    - `thinking`：显示脉冲/心跳态 + 可折叠 thinking 面板
    - `streaming`：正文逐 chunk 累积
    - `done`：转为正常 assistant message
    - `error`：转为错误态 assistant message
  - thinking UI 固定为：
    - 默认显示“思考中”状态条
    - 若收到 `thinking_delta`，显示可折叠 thinking 区
    - 若没有 `thinking_delta`，仍保留 pulse/heartbeat
  - citations UI 固定为：
    - 默认折叠
    - 回答主体优先展示，依据在回答下方的小面板里展开
    - 标签继续区分 `public_policy / private_sample / private_upload / carbon_factor`
  - 回答正文改为真实逐 chunk 渲染，不做额外花哨动画。
  - 消息角色视觉重新拉开：
    - `user`
    - `assistant`
    - `system`
    - `assistant(thinking/streaming)` 在生成态有独立样式
  - 当前 Ask 页顶部的 memory 状态条继续保留，但新增一条更用户可感知的 `context source` 摘要文案。

- 实施协同拆分：
  - Agent 1：后端 provider/orchestrator/SSE route/session message update
  - Agent 2：前端 Ask 流式状态机、thinking UI、citations 折叠、context source 呈现
  - Agent 3：测试、文档、`3rdparty/claw-code-study` 学习映射沉淀
  - 主线程只做集成、接口对齐和最终回归，不让三边写集冲突文件

## Public APIs / Interfaces
- 新增：
  - `POST /api/v1/sessions/{id}/ask/stream`
    - 请求体与现有 ask 相同：`question`、`knowledge_scope`、`top_k`、`attached_knowledge_item_ids/attached_file_ids`
    - 响应为 SSE
- 保留：
  - `POST /api/v1/sessions/{id}/ask`
    - 继续作为非流式兼容入口
- SSE 事件契约固定：
  - `event: message_start`
  - `event: status`
  - `event: thinking_delta`
  - `event: answer_delta`
  - `event: metadata`
  - `event: done`
  - `event: error`
- `GET /api/v1/sessions/{id}` 继续返回 `memory_state`，前端必须消费：
  - `context_usage_estimate`
  - `context_budget_estimate`
  - `summary_present`
  - `summary_updated_at`
  - `compacted_message_count`
  - `compaction_status`
- 不新增新的后台或知识流 API；`memory_notes` 继续只保留后端接口，不上前端导航。

## Assumptions
- 当前 `main` 与 `release/cloud-stable` 已经在 `v1.1.3`，所以本轮的 housekeeping 只做确认和文档同步，不再重复做大合并。
- 本地学习参考以 `3rdparty/claw-code-study/fix_ui_parity_snapshot` 为 runtime/streaming 主参考；配置治理参考 `3rdparty/claw-code-study/main_snapshot/CLAW.md`。

- 流式实现优先保证 ask 主链路和聊天体感，不顺带重做 report/calc 的流式化。
