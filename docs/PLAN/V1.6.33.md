# V1.6.33 RAG-Pro Parity Freeze and Handoff

## Summary
- 本轮冻结 `change-id: rag-pro-parity-freeze-and-handoff`，目标是把当前 RAG-Pro 主线从“持续施工态”整理成可交付 baseline。
- 只做 RAG baseline 冻结、文档同步、工作台入口语义、Milvus 生命周期证据、Workbench/AskPage 一致性验收；不做碳核算、知识图谱、新模型、主题美化。
- 已核验：`upload/status/run-pipeline/run-pipeline-batch`、`timing_trace`、`rag_pro_search/answer`、Test QA grounded answer 已存在；README 仍停在 V1.6.29，Milvus warm search 需要补更硬的复用测试和冻结文档。

## Key Changes
- 更新 `README.md` 顶部为 `V1.6.33 RAG-Pro Parity Freeze / RAG 主线进入 baseline 冻结`，新增 `docs/PLAN/V1.6.33.md`、`docs/research/rag/RAG_PRO_PERFORMANCE_GAP_AUDIT_V1.6.33.md`，并在 `RAG_RUNTIME_PROFILES.md` 写清 V1.6.32 已合并、版本判断以 `main` commit 为准。
- 新增 `docs/research/rag/RAG_PRO_PARITY_CHECKLIST_V1.6.33.md`，逐项标注“已完成 / 部分完成 / 未完成”，每项必须包含证据文件、测试名、剩余风险；冻结 #1 继续把关项与交给 #3 的体验项。
- 工作台按钮统一改名：单文档和批量默认入口叫“快速建立 RAG”，调用 `pipeline_mode=quick`，只跑 `parse -> chunk -> index -> search smoke`；完整入口叫“完整验收 RAG”，调用 `pipeline_mode=acceptance`，再跑 eval/可选 Test QA。
- Milvus 生命周期冻结：保留当前按 URI 缓存 `MilvusClient` 的实现，同时把 `get_vector_store()` 改为按 backend 复用 adapter 实例；补测试证明同一 URI 第二次 warm search `milvus_client_init_count=0`，并且 index 后 search 不重新初始化 client。
- Workbench/AskPage 一致性验收：用青木制造验收问题固定验证 Workbench search、Workbench Test QA、AskPage 三路都命中 `217,650 kWh`，且 `trace.kb_id` 与 citation 来源 KB 一致；CI 用 fake provider/memory backend，真实验收用 Docker Milvus + BGE-M3。

## Public Interfaces / Types
- 不新增新业务接口；继续使用现有 `/api/v1/kb/{kb_id}/documents/upload`、`/status`、`/run-pipeline`、`/run-pipeline-batch`、`/api/v1/rag/search`、`/rag/test-qa`、`/rag/answer`、`/rag/eval/run`。
- `pipeline_mode=quick|acceptance` 语义写死：quick 不调用大模型、不跑完整验收评分；acceptance 才允许 eval 和生成验证。
- `/api/v1/rag/retrieve`、`RagLabPage`、`langchain_rag_search/answer` 继续保留兼容，但文档和验收路径只认 `KnowledgeBaseWorkbench`、`AskPage`、`rag_pro_search`、`rag_pro_answer`。
