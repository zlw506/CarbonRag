# V1.6.4 RAG-Pro 真实向量与 Ask 主链路验收计划

> V1.6.8 更新：native Windows 不再使用 `milvus_lite + .db` 作为默认验收路线，改为 Docker Milvus Standalone：`RAG_VECTOR_BACKEND=milvus`、`RAG_MILVUS_URI=http://127.0.0.1:19530`。本文件中关于 Windows 本机 Milvus Lite 的描述仅保留为历史记录。

## Summary
- 本轮 `change-id: rag-pro-real-vector-ask-e2e`，目标是把 V1.6.3 的 RAG-Pro 脊柱从“结构已搭好”推进到“真实 Milvus Lite/BGE-M3 可跑、AskPage 可选择知识库、真实文档端到端可验收”。
- 当前 CarbonRag GitNexus 索引已 stale，实施第一步必须重建索引并用 GitNexus 做影响分析；RAG-Pro 与 ragPdfSystem 索引可继续作为参考。
- RAG-Pro 迁移准则继续有效：允许直接迁移/改写 RAG-Pro 核心结构与逻辑，但不提交 `3rdparty` 源码，迁移模块保留来源说明。

## Key Changes
- GitNexus 开工纪律：新建分支后先跑 `openspec validate --all`，再重建 CarbonRag GitNexus 索引；若 GitNexus 自动改写 `AGENTS.md/CLAUDE.md`，只保留人工确认的改动。实现前用 `gitnexus query/impact` 定位 `backend/app/rag`、Ask 链路、KB 工作台影响面，完成后用 `gitnexus detect_changes` 做复核。
- 真实向量底座：把 `backend/app/rag/vector_backend/milvus_store.py` 从占位实现改为 Milvus Lite/Milvus 持久化 adapter，默认本地 URI 为 `./data/outputs/milvus_lite/carbonrag.db`，collection metadata 必须包含 `kb_id/doc_id/chunk_id/owner_user_id/source_type/page/sheet/slide/title`。
- BGE-M3 接入：新增 RAG-Pro 风格 embedder，默认 `BAAI/bge-m3`，输出 dense + sparse；优先读取本地模型目录，失败再走 Hugging Face/HF mirror/proxy。模型不可用时返回 `degraded/error trace`，禁止用 hash embedding 或假向量显示 RAG 成功。
- Hybrid 检索闭环：实现 Milvus dense+sparse 查询、RRF 合并、真实 rerank adapter，默认 reranker 为 `BAAI/bge-reranker-v2-m3`；reranker 不可用时必须 `rerank_applied=false` 并给 warning，不允许静默伪成功。
- 文档索引闭环：`KnowledgeBase -> Document -> Chunk -> Milvus/BGE index` 成为主路径；V1.5.1 文件解析结果进入 KB document/chunk/index 状态机，`uploaded -> parsed -> chunked -> indexed -> failed` 必须可观测。
- Ask 主链路：`/sessions/{id}/ask` 与 `/ask/stream` 增加 `kb_id`、`rag_mode`，默认 `hybrid_rerank`；真实聊天必须调用 `backend/app/rag` 主入口并返回 retrieval trace，不再默认回退旧 `RagEngineService` 掩盖失败。
- 前端验收入口：AskPage 增加知识库选择、检索模式选择、RAG 状态条和 trace 展示；KnowledgeBaseWorkbench 增加直接上传/导入文档、解析/分块/索引进度、chunk 预览、test QA、vector backend health。
- 配置默认：`.env.example` 将比赛版默认固定为 `RAG_VECTOR_BACKEND=milvus_lite`、`RAG_EMBEDDING_PROVIDER=openai_compatible`、`RAG_RERANK_ENABLED=true`、`RAG_RERANK_PROVIDER=bge_reranker`、`RAG_REQUIRE_REAL_VECTOR=true`；`memory/chroma` 只作为开发或故障兼容模式。本地 BGE-M3 路线仍保留，但只在 smoke 或明确配置时启用。
- 本机默认不自动下载大模型：`RAG_MODEL_AUTO_DOWNLOAD=false`。日常开发可先用 `RAG_EMBEDDING_PROVIDER=openai_compatible` 走云端 embedding API；只有执行真实 smoke 时才显式开启 `RAG_MODEL_AUTO_DOWNLOAD=true`，并且必须把 `RAG_MODEL_CACHE_DIR/HF_HOME/HUGGINGFACE_HUB_CACHE` 指向空间充足的数据盘。

## Public APIs / Interfaces
- `POST /api/v1/kb/{kb_id}/documents/{doc_id}/index` 必须真实写入 Milvus Lite/Milvus，并返回 `vector_backend/indexed_chunk_count/degraded/warnings`。
- `POST /api/v1/rag/search` 必须返回 `dense_score/sparse_score/rrf_score/rerank_score/vector_backend/degraded/rerank_applied/kb_id/doc_id/chunk_id/citation metadata`。
- `POST /api/v1/rag/test-qa` 必须走同一套真实检索链路，不能绕过 Ask 主路径。
- `/api/v1/sessions/{id}/ask` 和 `/ask/stream` 请求体新增可选 `kb_id`、`rag_mode`；响应 metadata/source_summary 透出 `dense_count/sparse_count/merged_count/rerank_applied/vector_backend/degraded/warnings/citation_count`。
- 前端 service type 同步新增 `kb_id`、`rag_mode`、RAG trace 字段，AskPage 和 KnowledgeBaseWorkbench 共用同一套类型。

## 本机真实 smoke 注意事项
- 2026-05-10 首次运行 BGE-M3 smoke 时，FlagEmbedding/HuggingFace 默认把 `BAAI/bge-m3` 缓存写入 `C:\Users\yusu\.cache\huggingface\hub\models--BAAI--bge-m3`，在 C 盘空间不足时会导致下载失败甚至占满系统盘。该目录已人工确认后删除；后续不得让模型默认落到 C 盘。
- 同次排查发现 reranker 也曾把 `BAAI/bge-reranker-v2-m3` 缓存写入 `C:\Users\yusu\.cache\huggingface\hub\models--BAAI--bge-reranker-v2-m3`。该目录已人工确认后删除；后续 smoke 也必须把 reranker 缓存指向大盘。
- `scripts/rag-pro-real-vector-smoke.ps1` 已显式设置 `HF_HOME` 与 `HUGGINGFACE_HUB_CACHE` 到项目 `data/outputs/hf-cache`。其他团队必须把 BGE-M3 / reranker 模型缓存放到空间充足的磁盘，例如 `F:\Project\CarbonRag\data\outputs\hf-cache` 或自己机器的大容量数据盘。
- `RAG_MODEL_CACHE_DIR` 默认改为 `./data/outputs/models`，该目录属于本地运行产物，不进 Git。不要把模型放进 `backend/app/models` 或其他源码目录，更不要把模型文件提交到云端。
- 旧配置曾在从 `backend/` 目录启动时把模型误落到 `backend/backend/app/models/BAAI/bge-m3`。该未跟踪生成目录已删除，`.gitignore` 增加 `backend/backend/` 防止后续误提交；其他团队如遇到类似嵌套目录，应先确认其为本地模型/缓存产物再删除。
- 如果团队已有云端 embedding API，优先使用：
  - `RAG_EMBEDDING_PROVIDER=openai_compatible`
  - `EMBEDDING_API_BASE_URL=<你的 OpenAI-compatible embedding endpoint>`
  - `EMBEDDING_API_KEY=<你的 key>`
  - `EMBEDDING_MODEL=<你的 embedding model>`
  这条路径不会下载本地 BGE-M3，但仍然是“真实 dense vector”，不是 hash/fake embedding。BM25 sparse 仍由 RAG spine 自己提供。
- 只有需要验证本地 RAG-Pro BGE-M3 路线时，才切换：
  - `RAG_EMBEDDING_PROVIDER=bge_m3`
  - `RAG_MODEL_AUTO_DOWNLOAD=true`
  - `RAG_MODEL_CACHE_DIR=F:\Project\CarbonRag\data\outputs\models` 或其他大盘路径
  - `HF_HOME=F:\Project\CarbonRag\data\outputs\hf-cache`
  - `HUGGINGFACE_HUB_CACHE=F:\Project\CarbonRag\data\outputs\hf-cache\hub`
- `BAAI/bge-m3` 官方仓含 ONNX 大文件和 `imgs/*` 辅助文件；本轮 Python embedder 下载时忽略 `*.onnx`、`*.onnx_data`、`onnx/*`、`imgs/*`，只保留 PyTorch 推理所需文件，避免无意义下载和镜像源 403。
- 如 smoke 报 `No module named 'milvus_lite'`，说明只装了 `pymilvus` 还不够，需要安装 `milvus-lite`。当前 Windows pip 源没有可用 `milvus-lite` wheel，因此 Windows 本机不能直接完成 Milvus Lite 文件型 smoke；应使用 WSL/Linux、Linux CI、或把 `RAG_MILVUS_URI` 指向远程 Milvus 服务。`backend/requirements.txt` 已用 platform marker 避免 Windows 安装失败。
- 如 smoke 报 `No module named 'peft'`，说明 `FlagEmbedding` 运行依赖未补齐，需要安装 `peft`。该依赖已写入 `backend/requirements.txt`。

## Assumptions
- 本轮不继续做第三套 RAG facade；`backend/app/rag` 是唯一主入口，`backend/app/langchain_rag` 只允许作为过渡算法 adapter。
- 本轮不接 ragPdfSystem 的 Celery/RabbitMQ/MinIO，不做知识图谱 UI；这些只保留为后续企业增强路线。
- `docs/碳核算专项/` 仍不属于本轮提交范围。
- 若真实 BGE-M3 或 Milvus Lite 在本机无法跑通，需要先解决下载、代理、依赖或模型路径问题；不能用 memory/hash fallback 伪装验收通过。
