# V1.6.29 RAG-Pro 性能根因审计与运行时加固计划

## Summary
- 本轮 `change-id: rag-pro-performance-parity-audit-and-runtime-fix`，目标不是继续凭感觉调 UI，而是把 RAG 慢、卡、时好时坏的问题拆成可观测、可复现、可修复的运行时问题。
- 当前只读核验显示：`upload/status/run-pipeline/run-pipeline-batch` 后端已存在，README 已到 V1.6.24，RagLab 已 legacy/admin-only；真正高风险点是 Milvus client 每次新建、sparse 每次全量扫描、pipeline 默认做 eval smoke、缺少 timing trace/profile。
- GitNexus 当前 stale，实施时先重建；RAG-Pro 仍是主复现对象，允许直接迁移/改写核心结构与逻辑，但不提交 `3rdparty` 源码树。

## Key Changes
- 新增审计文档 `docs/research/rag/RAG_PRO_PERFORMANCE_GAP_AUDIT_V1.6.29.md`，逐项对照 CarbonRag 与 `3rdparty/RAG-Pro/RAG-Pro` 的 vector store、retriever、chunker、KB/document/chunk、Test QA 和前端工作台；记录已修、未修、仍属 legacy 的差异。
- 锁死 KB API 合约：确认并补测试覆盖 `documents/upload`、`documents/{doc_id}/status`、`run-pipeline`、`run-pipeline-batch`；前端 service 与后端 response model 必须一致，不允许 404/405 或假成功。
- Milvus runtime 改为复用：按 `rag_milvus_uri` 缓存 `MilvusClient`，按 collection 缓存 `has_collection` 结果；index/search/health 复用同一个 client，并在 trace 中暴露 `milvus_client_init_count`。
- 新增 `RagTimingTrace`：pipeline/search/answer 返回 `embedding_ms`、`milvus_client_ms`、`milvus_insert_ms`、`milvus_search_ms`、`db_load_chunks_ms`、`sparse_ms`、`rrf_ms`、`rerank_ms`、`llm_ms`、`total_ms`、各阶段候选数和 `loaded_chunk_count`。
- pipeline 拆成两档：默认 `quick_pipeline=parse→chunk→index→search_smoke`，不跑 eval/LLM；`acceptance_pipeline=parse→chunk→index→search_smoke→eval→optional_answer`，由工作台显式按钮触发。
- sparse 检索加缓存：按 `kb_id + chunk updated watermark` 缓存 token/corpus，按 query 缓存 tokens，只对当前 KB indexed chunks 建 corpus；trace 增加 `sparse_cache_hit`、`sparse_loaded_chunk_count`、`sparse_ms`。
- `/rag/answer` 保持正式 grounded answer：检索 chunks 后调用当前 provider；provider 不可用时返回 `answer_mode=retrieval_only`，前端明确显示“未调用大模型，仅展示检索片段”。

## Public APIs / Types
- `POST /api/v1/kb/{kb_id}/documents/{doc_id}/run-pipeline` 增加可选 `pipeline_mode=quick|acceptance`，默认 `quick`。
- `POST /api/v1/kb/{kb_id}/documents/run-pipeline-batch` 同步支持 `pipeline_mode` 和 `doc_ids`。
- `/api/v1/rag/search`、`/api/v1/rag/answer`、`/api/v1/rag/test-qa`、pipeline response 统一返回 `timing_trace`。
- 保留现有 `/rag/retrieve` legacy/admin-only，不进入验收路径；正式验收入口仍为 `/rag/search`、`/rag/answer`、`/rag/test-qa`、`/rag/eval/run`、KnowledgeBaseWorkbench、AskPage。

## Test Plan
- 后端新增/更新：KB API 合约测试、pipeline quick 不跑 eval/LLM、acceptance 才跑 eval、Milvus client index/search 复用、第二次 warm search `milvus_client_init_count=0`、sparse cache hit、`/rag/answer` provider 不可用时 `retrieval_only`。
- 性能脚本新增：`scripts/rag-profile-pipeline.ps1/.sh`、`scripts/rag-profile-search.ps1/.sh`，输出 `logs/rag/profile-v1.6.29.json`。
- 前端验证：工作台按钮改为“快速入库”和“验收评分”；trace 面板显示 timing、cache、Milvus init、候选数；慢点能定位到 embedding/Milvus/sparse/rerank/LLM。
- 回归命令：`openspec validate rag-pro-performance-parity-audit-and-runtime-fix --strict`、`openspec validate --all`、后端 targeted pytest、前端 `typecheck/build`、`git diff --check`、重建后 `gitnexus detect_changes`。

## Assumptions
- Docker Milvus、BGE-M3、reranker 是真实性能验收前置；若 Milvus 未启动或模型缺失，只记录为环境 blocker，不把 degraded/no-hit 误判为代码性能。
- 本轮先做同步 profile 和缓存，不引入 Celery/RabbitMQ/MinIO；长任务队列化留给后续 ragPdfSystem 企业增强阶段。
- 不做新算法、不做知识图谱、不做碳核算、不改主题视觉；只修 RAG-Pro 主链路性能、接口一致性和可观测性。
