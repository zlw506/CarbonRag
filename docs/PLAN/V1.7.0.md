# V1.7.0 Crawler-to-RAG Auto Update Baseline

## Summary
- 本轮 `change-id: crawler-to-rag-auto-update-baseline`：把现有 policy crawler 雏形接入 RAG-Pro KB quick pipeline，形成“官方 source -> crawl run -> candidate -> 发布到 RAG KB -> quick pipeline -> RAG/AskPage 可检索引用”的闭环。
- 参考项目只学习本地已有 `3rdparty/crawl4ai-main/crawl4ai-main` 和 `3rdparty/crawlab-main/crawlab-main`；不下载 Firecrawl，不引入完整 Crawlab，不用 Crawl4AI 全量替换 Scrapy。
- 保留当前 Scrapy/Scrapyd 执行器；新增 crawler-to-RAG bridge，把 candidate 发布到共享 KB“官方政策自动更新库”，并执行 quick pipeline。

## Key Changes
- 新增文档与 OpenSpec：`openspec/changes/crawler-to-rag-auto-update-baseline/`、`docs/architecture/CRAWLER_TO_RAG_AUTO_UPDATE_ARCHITECTURE.md`、`docs/research/crawler/CRAWL4AI_CRAWLAB_ADOPTION_NOTES.md`、`docs/PLAN/V1.7.0.md`。文档写清：Crawl4AI 学 markdown 清洗/缓存/链接抽取，Crawlab 学 source/run/task/log/result/schedule/worker 治理，RAG-Pro KB 是唯一正式入库目标。
- 改安全默认值：`RAG_POLICY_LIVE_CRAWLER_MANUAL_ENABLED=true`、`RAG_POLICY_LIVE_CRAWLER_SCHEDULED_ENABLED=false`、`RAG_POLICY_LIVE_CRAWLER_AUTO_PUBLISH=false`，保留 `RAG_ENABLE_POLICY_CRAWLER=true` 与 `RAG_POLICY_CRAWLER_BACKEND=local_scrapy`；更新现有 crawler 测试中对默认自动发布/定时的断言。
- 扩展 candidate artifact metadata：保留现有表结构，优先通过 `metadata` 保存 `canonical_url/http_status/raw_storage_path/cleaned_storage_path/markdown_storage_path/previous_content_hash/change_type/skip_reason/crawler_backend/robots_obey/duration_ms/error_stage/error_detail`。失败阶段统一使用 `fetch/discover_links/parse_html/download_binary/clean_markdown/topic_filter/dedupe/stage/publish_to_rag/rag_quick_pipeline`。
- 新增清洗产物约定：现有 `storage_path` 继续作为 raw artifact；新增 sibling artifact 目录 `backend/data/public/policy_crawl_artifacts/{run_id}/{candidate_id}/`，写入 `cleaned.txt` 与 `document.md`。本轮先移植 Crawl4AI 的 LLM-ready markdown 输出思想，不新增 Crawl4AI 运行时依赖。
- 新增 `backend/app/rag/kb/crawler_bridge.py`：实现 `publish_crawled_candidate_to_rag_kb(candidate_id, reviewed_by_user_id)`，读取 candidate，创建/复用共享 KB“官方政策自动更新库”，创建 `source_type=public_policy` 的 RagDocument，调用 `run_document_pipeline(..., pipeline_mode="quick")`，并回写 `metadata.rag_kb_id/rag_doc_id/rag_pipeline_status/indexed_chunk_count/rag_search_smoke_passed/vector_runtime/degraded/error_stage/error_detail`。
- 新增 admin 发布接口：`POST /api/v1/admin/policy-crawler/candidates/{candidate_id}/publish-to-rag`。旧 `/publish` 保留但标记为 legacy knowledge publish；新验收只看 `publish-to-rag`。`unchanged` candidate 默认不重复发布，除非此前没有 `rag_doc_id`。
- Admin 页面补完整可视化闭环：Crawler 状态、Sources、Run、Runs、Candidates、Publish to RAG、RAG KB/Doc/index 状态、search smoke 结果、失败阶段和错误详情。旧发布按钮降级为“旧知识条目发布”。
- 新增 smoke 脚本：`scripts/crawler-smoke-local-scrapy.ps1` 跑 status/seed/run/list candidates；`scripts/crawler-smoke-rag-publish.ps1` 发布首个 pending candidate 到 RAG，并用 `/rag/search` 搜“碳达峰”。

## Public APIs / Interfaces
- 新增 `POST /api/v1/admin/policy-crawler/candidates/{candidate_id}/publish-to-rag`，返回扩展后的 candidate summary，包含 `rag_kb_id`、`rag_doc_id`、`rag_pipeline_status`、`rag_indexed_chunk_count`、`rag_search_smoke_passed`、`rag_error_stage`。
- 旧 `POST /api/v1/admin/policy-crawler/candidates/{candidate_id}/publish` 保留兼容，但 UI/文档标记为 legacy，不作为 V1.7.0 验收入口。
- Candidate metadata 标准成为后续 crawler/run/result 的稳定接口；`change_type=unchanged`、`skip_reason=duplicate_content_hash` 的 candidate 不进入默认 RAG 发布路径。
- 共享 KB 默认 `visibility=shared`、`source_type=public_policy`，保证团队可检索，同时避免后续自动发布直接扩大到公开发布。

## Assumptions
- 本轮仍走同步 quick pipeline，不引入 Crawlab 的 MongoDB/SeaweedFS/master-worker，也不接 ragPdfSystem 的 Celery/MinIO。
- Scrapyd 是后续稳定运行推荐路线；本轮保留 `local_scrapy` 作为默认手动 smoke，`scrapyd` provider 继续可配置。
- 自动定时和自动发布明确后移到 V1.7.x；V1.7.0 只允许手动触发、人工 review 后发布到 RAG。
- Docker Milvus、BGE-M3、RAG-Pro quick pipeline 是真实验收前置；若向量库或模型不可用，只记录环境 blocker，不把 degraded 结果写成成功。
