# CarbonRag v0.2.0 Report Generation 首个真实闭环计划

## Summary
- 本轮把 report 从占位推进到真实链路：用户可在某个 `session` 下，基于已存在的 ask 消息与 citations、可选的一条 `calc-carbon` 结果，生成、保存、重开、编辑和重新生成报告。
- 已锁定两项实现方式：
  - 报告正文采用“模板驱动 + 受控 LLM”，不是纯拼接，也不是开放式大作文。
  - 报告支持“编辑并保存”；保存为覆盖当前报告正文，不做版本 diff。
- `calc-carbon` 现有链路已具备稳定基线，因此 v0.2.0 直接实现 3 类报告：
  - `policy_summary`
  - `mixed_analysis`
  - `carbon_summary`

## Key Changes
- 后端新增 `backend/app/report/`，由 `service + composer + templates + storage + renderer` 组成：
  - `service` 负责输入校验、session 绑定、来源解析、调用 provider、持久化与读取。
  - `composer` 从 selected assistant messages、dedup 后的 ask citations、可选 carbon result 组装 report context。
  - `templates` 固定 3 个模板的章节骨架与约束，不允许把模板散落在路由里。
  - `renderer` 将结构化章节渲染成 Markdown；provider 返回按章节组织的结构化内容，而不是整篇自由文本。
  - `storage` 负责 SQLite / PostgreSQL 双后端下的 `reports` 与 `report_sources` 持久化。
- 报告生成不走现有 `ai_runtime report mode`；本轮由 `ReportService` 直接复用当前 provider factory 生成文本，避免把 report 和 ask/runtime 一起重构。`report mode` 继续保留为后续统一入口的预留位。
- 运行时数据库新增两张表，并同时补齐 SQLite / PostgreSQL schema：
  - `reports`
  - `report_sources`
  - `reports` 额外保存 `citations_json` 与 `source_summary_json` 快照，保证报告重开时不依赖原消息再次计算。
- 每份报告必须绑定到 `session_id`；报告来源显式记录为：
  - `message`
  - `citation`
  - `carbon_result`
- 报告生成成功后，当前 session 自动追加一条 `system` 消息，包含 `report_id`、`report_type`、标题摘要。为此 session message role 从 `user | assistant` 扩展为 `user | assistant | system`。
- 报告类型规则固定为：
  - `policy_summary`：至少 1 条 assistant message，且所选消息里必须有 public policy citations
  - `mixed_analysis`：至少 1 条 assistant message，且所选消息里必须同时有 `public_policy` 和 `private_sample` citations
  - `carbon_summary`：必须提供 1 条 `carbon_result_id`；assistant messages 可选
- 报告生成默认来源选择：
  - 初次进入报告页时，默认选中当前 session 最近一条带 citations 的 assistant message
  - 切到 `carbon_summary` 时，如当前 session 有 calc 记录，默认选中最近一条 carbon result
- “重新生成”语义固定为：按当前配置再次 `POST` 生成一份新报告，不覆盖旧报告。
- “保存编辑”语义固定为：对当前打开的 `report_id` 执行 `PATCH`，直接覆盖 `title/content/updated_at`，不保留历史版本。
- 前端报告页从占位页升级为真实 workbench：
  - 左侧：当前 session 信息 + 当前 session 下的报告列表
  - 中间：报告配置区 + 预览/编辑区
  - 右侧：依据面板，按 `public_policy / private_sample / carbon_factor` 分组显示
- 报告页交互固定为：
  - 类型选择、标题输入、assistant message 多选、可选 carbon result 单选
  - `生成`、`重新生成`、`复制 Markdown`、`保存编辑`
  - 预览区默认展示 Markdown 渲染结果，并提供原始 Markdown 可编辑视图
- 前端新增一个轻量 Markdown 渲染依赖，例如 `react-markdown`；不引入富文本编辑器。
- 双环境策略保持不变：
  - `local-dev`：SQLite fallback，本地实验报告
  - `cloud-stable`：PostgreSQL，稳定可测报告
  - 更新 `README`、`docs/API_BOUNDARY_DRAFT.md`、`docs/deploy/LOCAL_DEV_VS_CLOUD_STABLE.md`、新增 `docs/architecture/REPORT_GENERATION_FLOW.md` 与 `docs/PLAN/v0.2.0.md`

## Public APIs / Types
- 新增主接口：
  - `POST /api/v1/reports`
  - `GET /api/v1/reports/{report_id}`
  - `PATCH /api/v1/reports/{report_id}`
  - `GET /api/v1/sessions/{id}/reports`
  - `GET /api/v1/sessions/{id}/carbon-calculations`
- 兼容性接口：
  - `POST /api/v1/generate-report` 保留为 deprecated alias，内部转发到 `POST /api/v1/reports`
- `POST /api/v1/reports` 请求体固定为：
  - `session_id`
  - `report_type: policy_summary | mixed_analysis | carbon_summary`
  - `title?`
  - `source_message_ids?: string[]`
  - `carbon_result_id?: string`
  - `output_format?: "markdown"`，本轮只接受 `markdown`
- `POST /api/v1/reports` / `GET /api/v1/reports/{id}` 返回完整 `ReportDetail`，至少包含：
  - `report_id`
  - `session_id`
  - `report_type`
  - `title`
  - `status`
  - `content`
  - `output_format`
  - `citations`
  - `source_summary`
  - `sources`
  - `trace_id`
  - `created_at`
  - `updated_at`
- `PATCH /api/v1/reports/{report_id}` 仅允许更新：
  - `title?`
  - `content`
- `GET /api/v1/sessions/{id}/reports` 返回 `ReportSummary[]`，至少包含：
  - `report_id`
  - `report_type`
  - `title`
  - `created_at`
  - `updated_at`
  - `source_count`
- `GET /api/v1/sessions/{id}/carbon-calculations` 返回当前 session 下可选的 calc 结果摘要，至少包含：
  - `trace_id`
  - `period_label`
  - `total_emission_kgco2e`
  - `created_at`
- 新增统一 `ReportCitation`，支持：
  - `source_type: public_policy | private_sample | carbon_factor`
  - `title`
  - `source`
  - `source_url | null`
  - `snippet`
  - `chunk_id | null`
  - `factor_id | null`
- HTTP 语义固定：
  - 输入/来源校验失败：`422`
  - session 或 report 不存在：`404`
  - provider 失败：`502`
  - 存储/渲染内部错误：`500`

  - `test_report_storage.py`

- 后端覆盖场景：
  - `policy_summary` 成功生成，并绑定 session
  - `mixed_analysis` 成功生成，并同时写入 public/private citations
  - `carbon_summary` 成功生成，并带入 calc result 与 carbon factor citations
  - 选中非法 `session_id` / `report_id` / `carbon_result_id` 返回正确错误码
  - `mixed_analysis` 缺少任一侧 citations 时返回 `422`
  - `PATCH /reports/{id}` 可保存编辑内容，重新读取后内容一致
  - 生成成功后 session 中新增 1 条 `system` 消息
- 前端 smoke：
  - 新建 session，连续问两轮后进入报告页
  - 默认选中最近一条带 citations 的 assistant message
  - 生成 `policy_summary`，能看到报告正文、trace、引用数量
  - 切回 session 再进入报告页，历史报告仍可打开
  - 如当前 session 有 calc 结果，再生成一份 `carbon_summary`
  - 编辑 Markdown 并保存后，刷新仍能读回修改后的内容
  - 右侧依据面板能区分 `public / private / carbon factor`
- 回归：
  - ask 的 `public / private_sample / mixed` 不退化
  - calc / feedback 不退化
  - 本地 SQLite 与云端 PostgreSQL 双模式都能保存并读取报告

## Assumptions
- 本轮开发分支按你的口径从当前本地最新业务线切出；不要求跟随仓库里已有 release/feature 命名混乱。
- `carbon_summary` 本轮直接实现，不再保留 stub，因为当前 calc 已有真实接口、持久化和测试基线。
- 报告默认标题规则：若未传标题，则使用“模板名 + session 标题”的组合标题。
- 本轮不做 docx/pdf 导出、不做版本历史、不做跨会话报告聚合、不做附件自动解析进报告。
