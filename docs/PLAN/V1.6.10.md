# V1.6.10 RAG-Pro Test QA 与 AskPage 绑定加固计划

## Summary
- 目标：把 RAG-Pro 主脊柱真正接牢到 AskPage 和知识库工作台，避免“页面看起来接了，业务链路没接牢”。
- 当前核验：当前 [sessions.py](F:/Project/CarbonRag/backend/app/api/v1/endpoints/sessions.py) 已把 `kb_id/rag_mode` 写入 `ChatRequest.payload`，但必须补回归测试锁死；当前 Test QA 仍是片段拼接式回答，不能冒充成熟 RAG 大模型问答。
- 正式入口冻结：AskPage 负责大模型对话；知识库工作台负责入库、检索、测试回答和 trace 验收；RagLab 降级为 admin-only legacy 实验入口。

## Key Changes
- AskPage 绑定加固：保留现有 `kb_id/rag_mode` 传递逻辑，但新增后端测试证明 `ChatRequest.payload`、AI Runtime tool arguments、retrieval trace 都使用选中的 KB 和检索模式；构造 KB-A/KB-B 隔离场景，选 KB-A 时不能命中 KB-B。
- Test QA 升级：`/api/v1/rag/test-qa` 改为“检索 + grounded prompt + 当前 chat provider 生成回答 + citations + selected_chunks + evidence_quality/confidence + retrieval_trace”，不写入 session 历史；无命中时返回明确 `no_hits`，不得让模型编答案。
- 工作台三按钮语义固定：`只测检索` 调 `/rag/search`；`生成测试回答` 调 `/rag/test-qa` 并显示是否调用大模型、provider、证据质量；`去 AskPage 正式对话` 跳转并带 `kb_id/rag_mode/question`。
- RAG 主线收敛： [spine.py](F:/Project/CarbonRag/backend/app/rag/spine.py) 是正式 RAG 主入口；`langchain_rag` 仅保留 adapter 兼容；`/rag/retrieve` 与 [RagLabPage](F:/Project/CarbonRag/frontend/src/pages/RagLabPage/index.tsx) 标记 legacy，RagLab 导航仅 admin 可见并改名为“Legacy RAG 实验台”。
- 失败透明化：若 trace 出现 `memory_dev/chroma_dev/legacy fallback/kb_id missing/no_hits`，前端必须显示为降级或失败，不得显示“RAG 成功”。

## Public APIs / Interfaces
- `POST /api/v1/rag/test-qa` 响应扩展：`answer_mode: "llm_grounded" | "retrieval_only" | "no_hits"`、`provider_name`、`model_name`、`selected_chunks`、`evidence_quality`、`confidence`、`retrieval_trace`、`citations`。
- `POST /api/v1/sessions/{id}/ask` 与 `/ask/stream` 不改字段名，但测试必须证明 `kb_id`、`rag_mode` 进入 runtime payload、tool arguments、response metadata/source summary。
- 前端类型同步：KnowledgeBaseWorkbench 的 Test QA 面板显示“大模型已调用/未调用”、证据质量、引用片段、trace；AskPage 继续显示 citation 与 RAG trace。

## Test Plan
- 后端新增或加固：`test_ask_page_selected_kb_is_passed_to_runtime`、`test_ask_page_rag_mode_is_passed_to_runtime`、`test_ask_with_two_kbs_only_retrieves_selected_kb`、`test_rag_test_qa_calls_chat_provider_with_grounded_context`、`test_rag_test_qa_no_hits_does_not_call_provider`。
- 前端验证：知识库工作台按钮文案清晰；Test QA 能显示 provider、evidence quality、selected chunks；RagLab 只 admin 可见；AskPage URL 参数能自动选中 KB 和检索模式。
- 回归命令：`openspec validate rag-pro-testqa-and-ask-binding-hardening --strict`、`openspec validate --all`、`cd backend && .\.conda\python.exe -m pytest`、`cd frontend && npm.cmd run typecheck && npm.cmd run build`、`gitnexus detect_changes -r CarbonRag`。

## Assumptions
- RAG 开发阶段仍允许完整迁移/改写 RAG-Pro 核心结构与逻辑，但不提交 `3rdparty` 源码树。
- Test QA 使用当前已配置的 chat provider；如果 provider 不可用，返回 provider error，并保留 retrieval trace，不退回片段拼接冒充大模型回答。
- 本轮不新增算法、不做知识图谱、不扩 Docker/Milvus 部署，只修 Ask 正式链路、Test QA 成熟度和 legacy 入口收敛。
