# V1.6.15 RAG-Pro Parity Evaluation and Hardening

## Summary
- 本轮 `change-id: rag-pro-parity-evaluation-and-hardening`，目标是把 CarbonRag 的 RAG-Pro 复现从“有骨架、有页面、有 trace”推进到“可上传、可索引、可评测、可定位失败”。
- 主实现对象仍然是 `3rdparty/RAG-Pro/RAG-Pro`，允许直接迁移/改写其核心结构与逻辑；保留 CarbonRag 的 auth、session、AskPage、file、report、citation、OpenSpec、GitNexus、Mattermost 边界。
- 当前 `docs/carbonrag_rag_acceptance_fixture_qingmu_2025q1.md` 是验收材料，需纳入 RAG fixture，不再停留为临时文档。

## Key Changes
- 新增 `docs/governance/RAG_PRO_ADOPTION_AUTHORIZATION.md`，写清 RAG-Pro 为 #2/tbx2835066135 组内授权主复现对象；RMA-MUN 只做算法参考，ragPdfSystem 只做企业增强蓝图。
- 扩展 RAG-Pro parity 字段：KB 增加 `embedding_model=BAAI/bge-m3`、`chunk_size=512`、`chunk_overlap=64`、`parent_chunk_size`、`rerank_top_n=5`、`retrieval_top_k=20`；Document 增加文件名、类型、大小、路径/文件关联、解析/分块/索引进度、错误阶段；Chunk 增加 `keywords`、`questions`、`milvus_id`、`parent_chunk_id`、`token_count`。
- 知识库工作台补“直接上传到当前 KB”链路：上传文件 -> 创建 RagDocument -> parse -> chunk -> index -> Test QA；UI 必须显示上传中、解析中、分块中、向量化中、已入库、失败原因和重试入口。
- `/rag/answer` 与 `/rag/test-qa` 统一为 grounded LLM answer：先检索，再构造证据提示词，再调用当前 chat provider，返回 `answer_mode`、`selected_chunks`、`citations`、`retrieval_trace`、`evidence_quality`、`confidence`；provider 不可用时只能返回 `retrieval_only`，前端必须明确“未调用大模型”。
- 新增正式 AI Runtime 工具名 `rag_pro_search`、`rag_pro_answer`；旧 `langchain_rag_search`、`langchain_rag_answer` 只保留兼容，不再写进主文档和验收路径。
- 新增 RAG 评测体系：`rag_eval_runs`、`rag_eval_cases`，并在工作台增加“验收评分”面板；把当前青木 2025Q1 fixture 放入 `backend/tests/fixtures/rag/`，新增 `gold_questions.json`。

## Public APIs / Interfaces
- 新增 `POST /api/v1/kb/{kb_id}/documents/upload`：上传文件到指定 KB 并返回 `doc_id`、状态、文件元数据。
- 新增 `GET /api/v1/kb/{kb_id}/documents/{doc_id}/status`：返回 parse/chunk/index 进度、失败阶段、错误信息、chunk/index 数量。
- 新增评测接口：`POST /api/v1/rag/eval/run`、`GET /api/v1/rag/eval/runs`、`GET /api/v1/rag/eval/runs/{run_id}`。
- `/api/v1/rag/answer` 响应稳定包含：`answer_mode`、`provider_name`、`model_name`、`selected_chunks`、`evidence_quality`、`confidence`、`citations`、`retrieval_trace`。
- `/api/v1/rag/search`、`/api/v1/rag/test-qa`、AskPage metadata 继续透出 `vector_runtime`、`dense_count`、`sparse_count`、`merged_count`、`rerank_applied`、`degraded`、`warnings`、`citation_count`。

## Test Plan
- 后端新增覆盖：KB 字段默认值、文档上传到 KB、Document 状态机、chunk keywords/questions/milvus_id、`/rag/answer` 调 provider、provider 不可用降级、`rag_pro_search/rag_pro_answer` 工具注册、跨 KB 隔离。
- 评测指标至少计算：Hit@1、Hit@3、Recall@5、Precision@5、MRR、citation_coverage、answer_mode_rate、no_hit_count、vector_failure_count、cross_kb_leak_count。
- 验收阈值固定：`vector_runtime=milvus_standalone`、`degraded=false`、文档 indexed 成功、Hit@3 >= 0.85、MRR >= 0.75、citation_coverage = 100%、cross_kb_leak_count = 0。
- 回归命令：`openspec validate rag-pro-parity-evaluation-and-hardening --strict`、`openspec validate --all`、`cd backend && .\.conda\python.exe -m pytest`、`cd frontend && npm.cmd run typecheck && npm.cmd run build`、`git diff --check`。
- GitNexus：当前 CarbonRag 索引 stale，实施前先重建索引；完成后用 `gitnexus detect_changes` 复核 RAG/API/frontend 影响面。

## Assumptions
- 本轮不做知识图谱 UI、不接 ragPdfSystem 的 Celery/RabbitMQ/MinIO、不扩碳核算业务。
- 允许直接迁移/改写 RAG-Pro 核心结构与逻辑，但不得提交 `3rdparty` 源码树。
- `docs/碳核算专项/` 仍不属于本轮提交；青木 2025Q1 RAG 验收 fixture 属于本轮。
- 如果 Milvus/BGE/provider 任一关键环节不可用，必须显示 degraded/failed，不能用 snippet 拼接或 memory/hash fallback 冒充 RAG 成功。
