# CarbonRag v0.0.2 稳定配置轮实施计划

## Summary
- 当前基线：仓库处于 `v0.0.1`，`main` 和 `dev` 已在本地与远程存在，前后端仍是空占位目录。
- v0.0.2 只做稳定工程壳，不进入 ask、calc、report 业务逻辑；施工分支固定为从已同步 `dev` 切出的 `feature/stable-bootstrap`。
- 已锁定实现默认值：后端运行时使用 `py -3.11`，后端依赖源使用 `requirements.txt`，前端包管理使用 `npm`，仓库启动入口使用 `scripts/bootstrap.ps1` 和 `scripts/bootstrap.sh`。

## Key Changes
- Git 与仓库基线：从 `dev` 拉齐远端后创建并推送 `feature/stable-bootstrap`，先做空的 `checkpoint: before v0.0.2`，最终提交固定为 `v0.0.2: establish stable bootstrap for frontend and backend`；同步更新 Git 工作流文档，明确 v0.0.2 不再重复创建 `dev`。
- 仓库级配置：新增 `.editorconfig`，统一 UTF-8、LF、末尾换行和全仓 4 空格缩进；保留根目录 `.env.example` 作为后端私有配置模板，新增 `frontend/.env.example` 且只允许 `VITE_API_BASE_URL`、`VITE_APP_TITLE`；补齐 `.gitignore` 对后端虚拟环境、前端依赖与构建产物、本地 env 文件的忽略；新增双启动脚本，负责复制 env 模板、安装前后端依赖、跑最小检查并打印下一步命令。
- 前端稳定壳：基于 Vite + React + TypeScript + Ant Design + React Router 建立可运行壳，提交锁文件；实现统一布局、四个可路由占位页、清晰目录结构，以及仅面向自家后端的服务层；当前前端只允许访问 `/healthz` 和 `/api/v1/system/info`，不得出现任何模型供应商 URL、模型密钥或页面直连第三方模型接口。
- 后端稳定壳：基于 FastAPI 建立最小服务，包含配置层、日志层、API 路由层、provider 抽象层，以及 rag/carbon/report/services/schemas 空壳包；实际只开放 `GET /healthz` 与 `GET /api/v1/system/info`；`system/info` 固定返回 `app_name`、`version`、`env`、`api_prefix`、`model_provider_mode`、`timestamp`；provider 只保留 base 抽象和 `cloud_api_stub` 元数据壳，不写真实供应商调用。
- 公共接口与文档契约：根目录后端 env 变量作为私有服务契约，前端 env 变量作为浏览器可见契约；新增开发启动文档和 API 边界草案；将 `/api/v1/ask`、`/api/v1/calc-carbon`、`/api/v1/generate-report` 只定义为 `draft only / not implemented`，写清输入草案、输出草案、错误码占位和“暂不鉴权”；README 状态推进为“第 0.0.2 轮稳定配置中”，写清前后端启动方式、当前仅开放的最小接口与仍未进入的业务范围；技术栈文档同步冻结 Python 3.11、FastAPI、React、Vite、TypeScript、Ant Design 与 provider 抽象策略。
- 数据与 CI：为 `data/public`、`data/private_sample`、`data/factors` 分别补 README，明确放什么和不放什么；`data/outputs` 继续只保留 `.gitkeep` 且不纳入运行结果版本控制；新增一个最轻量 GitHub Actions 工作流，在 `main`、`dev`、`feature/**` 的 push 和面向 `main`、`dev` 的 PR 上执行前端安装/构建与后端安装/测试。

## Test Plan
- 分支与提交：确认 `feature/stable-bootstrap` 跟踪 `origin/feature/stable-bootstrap`，并能清晰看到空 checkpoint 提交与最终结果提交。
- 前端：`npm install`、`npm run typecheck`、`npm run build`、`npm run dev` 均通过；四个页面可切换；全文检索前端源码不得出现 `MODEL_API_KEY`、`EMBEDDING_API_KEY` 或第三方模型 base URL。
- 后端：使用 `py -3.11` 创建虚拟环境并安装依赖后，服务能启动；`GET /healthz` 与 `GET /api/v1/system/info` 可访问；pytest 覆盖最小启动与两个接口；API 层不引入任何 ask/calc/report 业务实现。
- 启动脚本与文档：两套 bootstrap 脚本能在不覆盖现有本地 env 的前提下完成依赖安装和最小检查；README、启动文档、API 草案、数据目录说明与实际命令、目录和接口保持一致。
- CI：GitHub Actions 从干净 checkout 能通过，所用命令与本地文档一致。

## Assumptions
- 由于 `main` 与 `dev` 已在远程存在，v0.0.2 会从已同步的 `dev` 切出 `feature/stable-bootstrap`，不会机械重跑“新建 dev”命令。
- 后端统一以 `py -3.11` 为准；当前 `python` 默认指向 3.7.6，不作为本轮后端运行基线。
- 根目录 `.env.example` 继续承担后端模板角色；前端使用独立 `frontend/.env.example` 与本地 `frontend/.env`；本轮不再新增一份 `backend/.env.example`。
- 本轮不实现 ask/calc/report 真实业务、真实模型调用、真实企业数据接入，也不做会让占位页看起来像“功能已完成”的伪闭环。
