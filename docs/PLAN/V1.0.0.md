# CarbonRag V1.0.0

## Round Name
Identity, data isolation, and initial admin entry

## Goal
Move CarbonRag from a weakly isolated single-user prototype to a minimum multi-user enterprise trial system.

## Implemented Scope

### Authentication
- local username/password login
- open registration for normal users
- `HttpOnly` cookie-based server sessions
- logout
- forced first-login password change for seeded admin and reset-password flows

### Seed Admin
- initial account: `admin / 123456`
- created only when `users` is empty
- forced password change on first login
- recovery fallback: if the seed admin becomes unavailable, registering `admin / 123456` on the register page restores the initial admin account and forces password change on next login

### Data Isolation
The following resources are now user-owned:
- sessions
- reports
- feedback
- uploaded files
- private sample bindings
- carbon calculations

Cross-user reads return `404`.

### Roles
- `user`
- `admin`

Normal users can:
- use ask
- use calc
- use report
- upload files
- view only their own history

Admins can additionally:
- view system status
- manage users
- manage private sample entry state
- view feedback overview
- trigger knowledge refresh tasks

Admins do not read ordinary users' session or report body content through normal business APIs.

### Frontend
- `/login` page with login/register tabs
- forced password-change flow
- protected routes for existing workbench pages
- initial `/admin` page with:
  - user management
  - knowledge refresh overview
  - feedback overview
  - system connection status

## Not Included
- complex RBAC
- org/department hierarchy
- SSO
- audit dashboard
- advanced admin configuration center
- async task queue for refresh jobs

## Validation
- backend pytest passes with auth, admin, and isolation coverage
- frontend typecheck passes
- frontend production build passes
