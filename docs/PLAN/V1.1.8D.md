# CarbonRag V1.1.8C Thinking 保留 + 流式收尾平滑化

## Summary
- 本轮基于当前 V1.1.8B / V1.1.8C 实验分支继续做 Ask 流式体验收尾，不扩新业务。
- 要修的不是“有没有流式”，而是流式完成阶段的产品体验：
  - thinking 结束后不再瞬间消失
  - 最终回答不再依赖整轮回读 session 才变成最终格式
  - 对话区不再从头重建后再快速滑到目标回复
- 已锁定的实现方式：
  - thinking 内容要**会话内持久保留**
  - 结束时自动收起 thinking 区，但用户可再次展开查看
  - Ask 前端在 `done` 后采用**本地就地收口**，不再立即整轮 `loadSessionDetail()`
  - 最终 assistant 正文以 SSE `done/metadata` 给出的最终 answer 为准，并继续按 Markdown 渲染

## Key Changes
- 后端消息模型补齐 thinking 持久化：
  - `messages` 表新增 `thinking_content` 字段，SQLite / PostgreSQL schema 与迁移都补齐。
  - `backend/app/session/schemas.py` 与前端 `SessionMessage` 类型新增 `thinking_content?: string | null`。
  - `SessionStore.append_message()` / `update_message()`、SQLite/Postgres store 的 `SELECT / INSERT / UPDATE` 全部带上 `thinking_content`。
  - `SessionService.finalize_exchange()` 增加 `thinking_content` 参数；`begin_exchange()` 仍创建空 assistant 占位。
- 流式 ask 路由不再只保存最终 answer：
  - `POST /api/v1/sessions/{id}/ask/stream` 在结束时用 `RuntimeStreamHandle.state.thinking_fragments` 聚合真实 thinking 文本。
  - 只持久化 provider/orchestrator 真正返回的 thinking 内容；当前路由层那条 synthetic `thinking_delta` 只用于 UI 占位，不写库。
  - `metadata` 和 `done` 事件增加 `thinking_content`，让前端在不回读 session 的情况下拿到最终权威 thinking 文本。
  - 无真实 reasoning chunk 时，`thinking_content` 保持空；done 后只保留最终答案，不伪造“思考结果”。
- Ask 前端从“整轮回读 + 清 draft”改成“本地收口 + 后台轻同步”：
  - `frontend/src/pages/AskPage/index.tsx` 的 `streamDraft` 不再在 `done` 后立刻被 `loadSessionDetail()` 覆盖。
  - `onMetadata/onDone` 先把最终 `answer`、`thinking_content`、`citations`、`trace_id`、`source_summary`、`memory_state` 合并到当前 streaming assistant message。
  - 请求结束后把 draft 的 user/assistant 两条消息**本地提交进 `activeSession.messages`**，然后清掉 `streamDraft`。
  - `refreshSessions()` 仍保留，用于更新左侧会话标题/更新时间；但成功流式完成后不再立刻 `loadSessionDetail(activeSessionId)`。
  - 只有切换会话、显式刷新或重新进入页面时，才重新从后端拉完整 session；这时 thinking 内容因已持久化，仍可回看。
- Thinking 区从“临时态”改成“可回看但默认收起”：
  - 为 assistant message 增加受控展开状态，按 `message_id` 维护，而不是继续用每次渲染都重置的 `Collapse defaultActiveKey`。
  - `pending / thinking` 时默认展开。
  - `done / error` 时若当前展开，则自动收起；收起后仍可手动展开查看 `thinking_content`。
  - 有真实 `thinking_content` 时，完成态 label 从“思考中”切换为“思考过程”；没有真实内容则完成后不再显示 thinking 折叠块。
- 自动滚动从“每次重渲染 smooth scrollIntoView”改成“容器级追底”：
  - 去掉当前基于 `messageStreamEndRef.scrollIntoView({ behavior: "smooth" })` 的全量依赖。
  - 给 `chat-message-stream` 增加容器 ref 和“是否仍在底部附近”的跟踪逻辑。
  - streaming 中只有在用户仍靠近底部时才自动追底；若用户主动上滑查看旧消息，就不强制抢回滚动。
  - `done` 时执行一次最终底部对齐，但用容器 `scrollTop = scrollHeight` 或等效无动画方案，不再从顶部平滑滑到目标回复。
  - 本轮滚动控制只发生在消息流容器内，不触发页面级重排或整块消息列表重建。
- 最终回答格式固定为最终 answer 的 Markdown：
  - streaming 过程中仍允许按 chunk 增量渲染。
  - 收到 `metadata/done` 后，用最终事件里的 `answer` 替换流式累积正文，作为最终渲染源。
  - assistant 正文继续统一走 `ReactMarkdown`；若最终 `answer` 为空，则回退到累积的 streamed content。
  - 这保证“输出完毕后的最终显示格式”稳定，不再依赖后续整轮 session reload 才变“规范”。

## Public APIs / Interfaces
- 业务路由不新增，只扩现有 SSE / session 消息字段：
  - `GET /api/v1/sessions/{id}` 的 `messages[]` 新增 `thinking_content?: string | null`
  - `POST /api/v1/sessions/{id}/ask/stream` 继续沿用现有事件流，但 `metadata` / `done` 新增：
    - `thinking_content?: string | null`
- 不改：
  - `POST /api/v1/sessions/{id}/ask`
  - 现有 `status / thinking_delta / answer_delta / metadata / done / error` 事件名
- 前端行为契约更新为：
  - done 后 assistant thinking 默认收起但可再次展开
  - Ask 页不再通过整轮 session reload 来完成最终回答落地

## Test Plan
- 后端：
  - SQLite / PostgreSQL schema 测试覆盖 `messages.thinking_content`
  - `ask/stream` 在 provider 有 reasoning chunk 时：
    - `metadata/done` 带 `thinking_content`
    - `finalize_exchange()` 把 thinking_content 写进 assistant message
  - provider 无 reasoning chunk 时：
    - 仍有 thinking 状态事件
    - 最终 `thinking_content` 为空，不写 synthetic thinking
  - `GET /api/v1/sessions/{id}` 返回的 assistant message 可带 `thinking_content`
- 前端：
  - 发送后 assistant 占位立即出现，thinking 展开
  - 有 reasoning chunk 时，done 后 thinking 自动收起，但可重新展开
  - 无 reasoning chunk 时，done 后不残留伪造思考内容
  - 最终回答在 done 后直接停留在最终渲染状态，不再先消失再靠 session reload 回来
  - 切换会话再切回后，thinking 内容仍可回看
  - 手动上滑查看旧消息时，streaming 不强制抢回滚动；停留在底部时则正常追底
- 回归：
  - `npm.cmd run typecheck`
  - `npm.cmd run build`
  - 后端流式问答相关 pytest
  - 手工验证：发送一条有 reasoning 的消息，观察 thinking 收口、最终 Markdown、完成后滚动停留是否符合目标

## Assumptions
- 本轮继续基于当前实验分支推进，不要求立即合并到 `main`。
- “会话内持久保留”指：重进该会话后仍能查看同一条消息的 thinking 内容；不要求单独做 thinking 历史页或额外导出。
- synthetic thinking 只用于生成中占位，不算真实“思考结果”。
- 本轮不重写整个 streaming 协议、不改 report/calc/knowledge 页，只修 Ask 的 thinking 保留与流式收尾体验。
