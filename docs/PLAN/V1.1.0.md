# CarbonRag V1.1.0 私有知识库自动更新 + 管理后台主线

## Summary
- 本轮把当前“仓库样例 + session 挂接 + 手动刷新”的 private 流程，推进成真正的私有知识流：上传文件和被监控的私有样例目录都会进入统一的知识条目与任务流，并可被 ask / mixed 实际检索使用。
- 已锁定的产品决策：
  - 采用“双层知识库”：`个人知识库 + 管理员共享知识库`
  - 自动更新最小版同时覆盖：`上传入库 + 目录变动检测`
  - 任务执行采用：`应用内轻量后台任务`
  - 文件接入按“全量文档入任务流”执行；但只做文本提取，不做 OCR，无法提取正文的扫描件或不可读旧格式文件进入 `parse_failed` 并支持重试
- 本轮继续保持双环境：
  - `local-dev`：SQLite fallback，实验知识数据
  - `cloud-stable`：PostgreSQL，稳定验证知识流

## Key Changes
- 知识库数据模型与任务流：
  - 新增 `knowledge_items`、`knowledge_chunks`、`knowledge_tasks` 三类运行时表，并同时补齐 SQLite / PostgreSQL schema。
  - `knowledge_items` 统一承接两类来源：
    - 用户上传形成的个人知识条目
    - 由 `data/private_sample/` 导入或监控得到的管理员共享知识条目
  - 每个知识条目至少固定这些核心属性：
    - `knowledge_item_id`
    - `owner_user_id | null`
    - `library_scope = personal | shared`
    - `source_type = uploaded_file | private_sample_repo`
    - `title`
    - `mime_type`
    - `storage_path`
    - `parse_status`
    - `ingest_status`
    - `index_status`
    - `is_enabled`
    - `session_attachable`
    - `source_hash`
    - `source_mtime`
    - `last_error`
    - `updated_at`
  - `knowledge_tasks` 固定承接 `upload_ingest / rebuild / rescan / retry`，状态至少为：
    - `queued`
    - `running`
    - `succeeded`
    - `failed`
  - 任务执行流程固定为：
    - 发现来源
    - 生成/更新知识条目
    - 文本提取
    - 切块
    - 写入 `knowledge_chunks`
    - 标记索引完成
    - 失效相关 retriever 缓存
  - 轻量后台任务采用应用内单 worker runner：
    - FastAPI 启动时拉起
    - 顺序消费 `queued` 任务
    - 进程重启时把卡住的 `running` 任务回收为可重试/重新排队
    - 不引入 Celery、队列服务或独立 worker
  - 上传文件不再只是 session 附件：
    - 仍保留 `files` 记录和 session 绑定
    - 上传成功后自动创建个人 `knowledge_item` 并入队 ingest task
  - 原有 repo `private_sample` 样例在 V1.1.0 启动/迁移时导入为“管理员共享知识条目”，并保留现有 `is_enabled / session_attachable` 管理语义。
  - 目录自动更新最小版采用“指纹检测 + 待更新标记”：
    - 对被登记的共享私有样例源记录 `mtime + hash`
    - 后台定时轻扫与管理员手动扫描都会比较差异
    - 变动条目标记为 `pending update`
    - 管理员一键触发重建后进入 `knowledge_tasks`

- 检索、session 与 ask/mixed 接入：
  - `private_sample` / `mixed` 不再直接读仓库样例文本，而是改为读已索引的 `knowledge_chunks`。
  - `public` 仍保持现有公共政策检索流。
  - private/mixed 检索范围固定为“当前 session 已挂接的知识条目”：
    - 个人知识条目
    - 管理员共享知识条目
  - 现有 `session_private_samples` 升级为基于 `knowledge_item_id` 的 session 挂接模型；已有旧 doc_id 绑定在迁移时映射到导入后的共享知识条目。
  - ask / mixed 的 private 侧 citation 扩展为可区分：
    - `public_policy`
    - `private_sample`
    - `private_upload`
    - 并新增 `knowledge_item_id` 与 `library_scope`
  - mixed 继续保留 public + private 混合逻辑，但 private 部分改为来自已挂接知识条目，而不是裸仓库样例。

- 后端 API 与权限：
  - 新增用户侧知识库接口：
    - `GET /api/v1/knowledge-items`
    - `GET /api/v1/knowledge-items/{knowledge_item_id}`
    - `GET /api/v1/knowledge-tasks`
    - `POST /api/v1/knowledge-tasks/{task_id}/retry`
    - `PUT /api/v1/sessions/{id}/knowledge-items`
    - `GET /api/v1/me/uploads`
    - `GET /api/v1/me/reports`
    - `GET /api/v1/me/feedback`
  - 新增管理员主线接口：
    - `GET /api/v1/admin/knowledge-items`
    - `PATCH /api/v1/admin/knowledge-items/{knowledge_item_id}`
    - `GET /api/v1/admin/knowledge-tasks`
    - `POST /api/v1/admin/knowledge-tasks/scan`
    - `POST /api/v1/admin/knowledge-tasks/rebuild`
    - `POST /api/v1/admin/knowledge-tasks/{task_id}/retry`
  - 管理员页继续保留：
    - 用户列表
    - 反馈概览
    - 系统状态
    - 并升级知识区为真正的“知识条目列表 + 更新任务列表”
  - 权限边界固定：
    - 普通用户只能看自己的个人知识条目、自己的上传、自己的报告、自己的反馈
    - 普通用户可浏览管理员共享知识条目目录并挂接到自己的 session
    - 管理员可管理共享知识条目和所有任务状态
    - 管理员对普通用户个人知识仅看元数据和任务状态，不直接查看正文内容

- 文件解析支持矩阵：
  - 本轮任务流接收当前上传白名单中的全部格式：`pdf / doc / docx / txt / md / csv / xls / xlsx`
  - 解析策略固定为：
    - `txt / md / csv / xls / xlsx / docx / 可提取文本的 pdf`：进入正文提取与切块
    - `扫描型 pdf / 不可读旧二进制 doc 等无法提取文本的文件`：进入任务流，但落为 `parse_failed`，前端和后台可见失败原因并允许重试
  - 不做 OCR，不做图片文档识别，不引入复杂内容清洗平台

- 前端工作台与后台：
  - `/admin` 从入口页升级为真实后台主线，至少 5 个区块：
    - 用户列表
    - 知识条目/文档列表
    - 更新任务列表
    - 反馈列表
    - 系统运行状态
  - 新增普通用户“我的知识库”页，建议独立路由 `/my-knowledge`，至少包含 4 个标签页：
    - 我的上传
    - 我的样例/知识条目
    - 我的报告
    - 我的反馈
  - Ask 页不再继续塞更多管理按钮，只保留：
    - 当前 session 挂接知识条目的选择入口
    - 当前 scope 状态
    - 当前挂接数量与任务状态摘要
  - 前端对知识条目与任务至少要能展示：
    - 解析状态
    - 入库状态
    - 索引状态
    - 最近错误
    - 重试按钮
    - 共享/个人标识
    - 是否可挂接
  - 管理后台可手动触发：
    - 扫描目录变动
    - 重建待更新条目
    - 重试失败任务

## Public APIs / Types
- 新增用户侧核心资源：
  - `knowledge_items`
  - `knowledge_tasks`
- `GET /api/v1/knowledge-items` 返回当前用户可见的知识条目列表：
  - 个人条目 + 管理员共享条目
  - 支持按 `library_scope / source_type / parse_status / index_status / session_attachable` 过滤
- `PUT /api/v1/sessions/{id}/knowledge-items` 采用整组替换语义，绑定当前 session 可用的知识条目集合。
- `GET /api/v1/knowledge-tasks`：
  - 普通用户只看自己触发或属于自己知识条目的任务
  - 管理员通过 `/api/v1/admin/knowledge-tasks` 看全局任务
- 新增统一知识条目摘要类型，至少包含：
  - `knowledge_item_id`
  - `title`
  - `library_scope`
  - `source_type`
  - `parse_status`
  - `ingest_status`
  - `index_status`
  - `is_enabled`
  - `session_attachable`
  - `updated_at`
- ask citation 扩展：
  - `source_type: public_policy | private_sample | private_upload`
  - `knowledge_item_id?: string`
  - `library_scope?: personal | shared`
- 既有 ask / mixed / files / reports / feedback 接口继续保留，但 private 相关结果改从知识条目系统读取。

## Assumptions
- V1.1.0 继续沿用当前认证与数据隔离模型，不新增更复杂 RBAC、组织树或 SSO。
- 共享私有知识库只由管理员维护；普通用户不能把自己的个人知识条目直接提升为共享条目。
- 本轮不做 OCR、图片文档识别、复杂异步任务平台、自动爬虫、跨会话长期 memory。
- 现有 `knowledge_refresh_tasks` 不再作为主任务流使用；V1.1.0 以新的 `knowledge_tasks` 为唯一知识任务事实源。
- 旧有 repo private sample 与 session 绑定数据在迁移时统一映射到共享知识条目，避免现有演示数据失效。
