# CarbonRag v0.1.5B ask-mode 首个受控联通轮计划

## Summary
- 从 `feature/ai-runtime-architecture-freeze` 切出 `feature/ask-mode-controlled-link`，先提交 `checkpoint: before v0.1.5b`，最终提交 `v0.1.5b: wire first controlled ask-mode route through ai runtime`。
- 本轮只打通一条真实链路：前端问答页 -> `POST /api/v1/ask` -> `ai_runtime` ask mode -> `orchestrator.run()` -> `providers.factory` -> 当前 OpenAI-compatible API -> 返回答案。
- 已锁定两项实现口径：
  - 错误语义采用 `422/502` 分流：输入非法返回 `422`，provider 失败返回 `502`，且 `502` body 仍是结构化 ask 响应。
  - ask mode 本轮不调任何业务工具 stub，`allowed_tools` 和默认调度序列都置空，避免制造检索假象。

## Key Changes
- 后端公开路由与 API 契约：
  - 新增 `backend/app/api/v1/endpoints/ask.py` 并接入 `api/v1/router.py`，只新增 `POST /api/v1/ask`，不新增任何其他公开路由。
  - 新增公开 wire schema，例如 `backend/app/schemas/ask.py`，固定 `AskRequest` 与 `AskResponse`：
    - `AskRequest`: `question`、`knowledge_scope`、`top_k`
    - `AskResponse`: `answer`、`mode`、`status`、`citations`、`trace_id`
  - 路由层只做最小输入校验与 internal mapping：把 `question` 映射到 `ChatRequest.user_input`，把 `knowledge_scope`/`top_k` 放进 `payload`，固定 `mode="ask"`；路由层不得直接调 provider、拼 prompt 或写工具逻辑。
  - 输入校验策略固定为：
    - `question` 去首尾空白后不能为空
    - `question` 设长度上限并在 route/guard 双层保护
    - `knowledge_scope` 对外仍保留 `public | private_sample | mixed`
    - `public` 正常通过，`mixed` 内部降级为 `public`，`private_sample` 在本轮返回 `422`
    - `top_k` 接收但本轮不生效，默认 `5`
- `ai_runtime` 最小真联通升级：
  - ask mode 从骨架推进为最小可执行模式：`mode="ask"`，`allowed_tools=()`，`default_stub_tool_sequence=()`，prompt policy 明确“单轮通用双碳问答、不伪造引用、不声称已检索”。
  - `context_builder` 改为 ask 专用最小上下文，至少产出：
    - 系统角色说明：CarbonRag 是双碳问答 MVP
    - 当前模式：ask
    - 当前限制：未接 RAG、未接企业私有数据、不可伪造引用
    - 用户问题正文
    - `knowledge_scope_requested` / `knowledge_scope_effective`
  - `guards` 增加 ask 轮次校验：
    - `mode` 必须是 `ask`
    - `question` 不能为空且长度合规
    - 工具白名单为空时必须允许空工具序列正常通过
    - 禁止内部异常原样外泄
  - `orchestrator.run()` 升级为真实 ask 链路：
    - ask 成功时返回内部 `RuntimeResult(status="ok")`
    - provider 异常时捕获并返回 `RuntimeResult(status="provider_error")`
    - 不再为 ask 路径使用 `stub_ready`
  - `ChatResponse` / `RuntimeResult` 的内部状态字面量扩展为兼容 ask 真实链路，同时保留后续 calc/report 的 stub 状态。
- Provider 真调用：
  - 不新增第二套 provider 结构，只升级现有 `app.ai_runtime.providers.chat_openai_compatible`。
  - 基于现有 `httpx` 依赖实现真实非流式 OpenAI-compatible 调用，固定使用 `/chat/completions`。
  - provider 只从现有 `.env` / `backend/local_provider_config.json` 读取 `MODEL_*` 配置；不在业务层拼 `base_url`、`api_key`、`model`。
  - provider 输入固定为两条 message：
    - `system`: ask mode 角色与限制说明
    - `user`: 用户问题
  - provider 统一返回解析后的文本内容和最小 metadata；网络失败、超时、非 2xx、返回体缺字段都落到受控错误，不抛 traceback 给前端。
- 前端问答页最小实装：
  - 只改 `frontend/src/pages/AskPage` 及其服务/类型，其他三页继续保持占位。
  - 新增 ask 服务与类型，例如 `frontend/src/services/ask.ts`、`frontend/src/types/ask.ts`，前端只调用自家 `POST /api/v1/ask`。
  - 问答页改为最小可用表单：
    - 输入问题
    - 提交按钮
    - 固定发送 `knowledge_scope: "public"`、`top_k: 5`
    - 展示 `answer`、`status`、`mode`、`trace_id`
    - `citations` 为空时显示明确占位说明
    - `422` 与 `502` 都以页面内错误提示展示，不崩溃
  - 同步更新问答页标题/导航文案，明确 ask 页已联通，其他页仍是占位。
- 文档同步：
  - `README.md` 状态推进为“第 0.1.5B 轮 ask-mode 首个受控联通中”，写清 `POST /api/v1/ask` 已开放，`calc-carbon` / `generate-report` 仍未实现，当前问答未接 RAG/引用。
  - `docs/API_BOUNDARY_DRAFT.md` 中 `POST /api/v1/ask` 状态推进为 `controlled-linked / implemented-minimal`；明确当前仅支持单轮问答、`citations` 可为空、未承诺检索依据、未接 RAG。`calc-carbon` 和 `generate-report` 保持未实现。
  - 新增 `docs/PLAN/v0.1.5B.md`，写清本轮只做 ask、为何不做 RAG、为何不做 calc/report、验收标准。

## Public Interfaces / Types
- 新增公开路由：`POST /api/v1/ask`
- 公开请求体固定为：
  - `question: string`
  - `knowledge_scope: "public" | "private_sample" | "mixed"`
  - `top_k: number`
- 公开成功/失败响应统一为 ask 响应形状：
  - `answer: string`
  - `mode: "ask"`
  - `status: "ok" | "provider_error" | "invalid_input"`
  - `citations: []`
  - `trace_id: string`
- HTTP 语义固定：
  - 输入非法：`422`
  - provider 失败：`502`
  - 成功：`200`
- 内部契约继续保持：
  - 所有 ask 请求必须经过 `app.ai_runtime.runtime.orchestrator.run()`
  - 所有模型访问必须经过 `app.ai_runtime.providers.factory`
  - ask 路由对外返回 `AskResponse`，不直接暴露完整 `RuntimeResult`

## Assumptions
- 继续以现有 OpenAI-compatible API 和默认模型 `gpt-5.4` 为唯一上游，不新增模型切换 UI 或多 provider 策略。
- `backend/local_provider_config.json` 继续作为本机 chat provider 覆盖入口；本轮不调整 embedding provider。
- ask mode 本轮不做真实检索、不接知识库、不接企业样例，因此 `citations` 固定为空数组，`mixed` 只做内部降级而不产生混合检索行为。
- 本轮保持同步 FastAPI 路由和同步 provider 调用实现，避免为首条真实链路额外引入 async 改造。
