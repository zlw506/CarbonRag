# CarbonRag v0.1.4 AI Runtime 架构冻结与后端骨架计划

## Summary
- 本轮从 `feature/claw-code-study-baseline` 切出 `feature/ai-runtime-architecture-freeze`，先提交 `checkpoint: before v0.1.4`，最终提交 `v0.1.4: freeze ai runtime architecture and land backend skeleton`。
- 目标是把 v0.1.3B 的学习结论转译成 CarbonRag 自有 `ai_runtime` 五层骨架，并同步冻结架构文档；不新增对外 API 路由，不实现真实 ask/calc/report 业务，不改前端页面。
- 已锁定两项实现取舍：`backend/app/providers` 直接移除，不保留兼容别名；v0.1.4 不新增任何对外 stub 路由，AI Runtime 只通过内部服务层和单元测试自证。

## Key Changes
- 仓库与文档冻结：
  - 新增 `docs/architecture/AI_RUNTIME_ARCHITECTURE_FREEZE.md`、`AI_RUNTIME_EXECUTION_FLOW.md`、`AI_RUNTIME_TOOL_BOUNDARY.md`、`CLAW_CODE_TO_CARBONRAG_MAPPING.md`。
  - 更新 `docs/API_BOUNDARY_DRAFT.md`：`/api/v1/ask`、`/api/v1/calc-carbon`、`/api/v1/generate-report` 保持未实现，但每个接口补上“未来由 ai_runtime 驱动”的说明，并统一标记为 `stub-ready / not implemented`。
  - 更新 `README.md` 与 `docs/PLAN/v0.1.4.md`：项目阶段推进到“AI Runtime 架构冻结轮”，文档入口加入 `docs/architecture/`。
- 后端结构落地：
  - 新增 `backend/app/ai_runtime/`，固定为 `config.py`、`providers/`、`runtime/`、`tools/`、`modes/`、`schemas/` 六层。
  - 直接删除旧 `backend/app/providers/`，并把现有 `system.py` 的 provider 依赖切到 `app.ai_runtime.providers.factory`。
  - `app/core/config.py` 继续承担全局 Settings；`app/ai_runtime/config.py` 只做 ai_runtime 视角的配置封装，不新增环境变量，统一消费现有 `MODEL_*`、`EMBEDDING_*`、数据目录配置和本地 `backend/local_provider_config.json`。
- Provider 层冻结：
  - `providers/base.py` 定义 `BaseChatProvider`、`BaseEmbeddingProvider`、descriptor/result 协议。
  - `chat_openai_compatible.py` 与 `embedding_openai_compatible.py` 分离聊天与嵌入 provider；本轮只保证实例化、描述信息和最小 stub 能力，不要求真实模型响应闭环。
  - `factory.py` 提供唯一入口：`get_chat_provider()`、`get_embedding_provider()`；所有未来模型访问必须经 factory，不允许业务层直接读 env 或直接 new provider。
- Runtime / tools / modes / schemas 骨架：
  - `runtime/orchestrator.py` 提供内部统一入口 `run(request) -> RuntimeResult`，固定流程为：mode resolve -> context build -> guards -> stub tool invoke -> response format。
  - `context_builder.py` 返回结构化占位上下文，至少预留 `policy_context`、`enterprise_context`、`carbon_context`、`report_context`、`session_state`、`memory_slot`。
  - `guards.py` 只做三件事：模式白名单、工具白名单、明确拒绝 shell / 任意文件写 / 任意网页抓取 / 任意系统调用。
  - `tools/registry.py` 注册 5 个业务 stub 工具：`policy_retrieve`、`enterprise_retrieve`、`carbon_factor_lookup`、`carbon_calc`、`report_draft`；重复注册报错，未知工具解析失败。
  - `modes/ask_mode.py`、`carbon_mode.py`、`report_mode.py` 固定各自的 `mode_name`、`allowed_tools`、`default_stub_tool_sequence`、`response_schema`、`prompt_policy`。
  - 推荐默认工具集合：
    - `ask`：`policy_retrieve`、`enterprise_retrieve`
    - `carbon`：`enterprise_retrieve`、`carbon_factor_lookup`、`carbon_calc`
    - `report`：`policy_retrieve`、`enterprise_retrieve`、`report_draft`
  - `schemas/chat.py`、`tool.py`、`result.py` 固定最小内部结构：
    - `ChatRequest`：`mode`、`user_input`、`payload`、`trace_id`
    - `ToolCall` / `ToolResult`：工具名、参数、状态、输出、元数据
    - `ChatResponse` / `RuntimeResult`：模式、状态、上下文摘要、tool results、统一 response
- 现有接口与版本同步：
  - `GET /api/v1/system/info` 保持公开字段稳定，只改为从 ai_runtime factory 取 provider descriptor；版本推进到 `v0.1.4`。
  - 不新增 `/api/v1/ask`、`/calc-carbon`、`/generate-report` 路由；这些仅在 API 草案和内部 schema/mode 层进入 `stub-ready` 状态。

## Public Interfaces / Types
- 不新增任何新的公开 HTTP 路由。
- 新增的内部后端契约固定为：
  - `app.ai_runtime.providers.factory` 是唯一模型入口；
  - `app.ai_runtime.runtime.orchestrator.run()` 是唯一内部 AI Runtime 入口；
  - `ChatRequest`、`ToolCall`、`ToolResult`、`ChatResponse`、`RuntimeResult` 是后续 ask/calc/report 共用的最小骨架类型。
- `docs/API_BOUNDARY_DRAFT.md` 要明确三条未来接口与内部 `ai_runtime` 的映射关系，但继续保持“未实现”。

## Assumptions
- v0.1.4 只冻结 AI Runtime 架构与后端骨架，不做真实 RAG、Embedding 入库、真实 ask/calc/report、插件系统、真实 memory、前端业务重构。
- `app/ai_runtime/config.py` 复用现有 `.env` 键，不新增环境变量；本地 `backend/local_provider_config.json` 继续只作为 chat provider 的本机覆盖，embedding provider 默认走 `.env`。
- state / memory 在本轮只通过 `context_builder` 的结构化占位字段和架构文档保留位置，不新增单独实现包。
- 由于本轮不新增对外路由，AI Runtime 的“可运行”定义是：单元测试可通过、`system/info` 可继续运行、内部 orchestrator 能稳定返回 stub `RuntimeResult`。
