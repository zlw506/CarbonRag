# V1.6.3 RAG-Pro 主脊柱迁移与四项目纵向审查

## Summary
- 本轮 `change-id: rag-pro-core-spine-port`，目标是把 CarbonRag 的 RAG 主线从“旧 RAG + langchain_rag 过渡模块”重构为 RAG-Pro 式知识库系统。
- 当前实施状态：已新增 RAG-Pro 风格 `backend/app/rag` 主脊柱、KB/Document/Chunk 存储、RRF hybrid 检索、test QA、`/api/v1/kb`、`/api/v1/rag/search|answer|test-qa`，并让现有 `langchain_rag_search/answer` 工具转接统一主入口。
- GitNexus 状态：CarbonRag 当前 commit 索引已刷新；`detect_changes` 将本轮判定为 critical，符合 RAG API、runtime schema、前端导航与工具层同时变更的风险面。新建未跟踪文件在提交/重建索引前不会出现在 GitNexus symbol graph 中。
- 主复现对象固定为 `RAG-Pro`；`RMA-MUN` 只保留算法参考；`ragPdfSystem` 只作为异步、对象存储、评测、GraphRAG 后续蓝图。

## Key Changes
- 参考资产与治理：
  - 整理本地 ignored 参考目录到 `3rdparty/rag-study/`，保留现有原目录不删除。
  - 新增第三方采用说明，明确 `RAG-Pro/ragPdfSystem` 为组内授权可移植，`RMA-MUN` 未确认许可证只参考架构。
  - 新增四项目审查文档：四维 benchmark、RMA-MUN gap audit、RAG-Pro port map、ragPdfSystem enterprise blueprint。
- RAG 主入口重构：
  - `backend/app/rag` 成为唯一 RAG 主入口。
  - `backend/app/langchain_rag` 降级为 algorithm adapter；不得再作为新的并行主线。
  - 禁止生产路径使用 `hash_embedding` 假向量显示成功；真实 embedding/vector 不可用时必须返回 degraded/error trace。
- RAG-Pro 核心能力迁移：
  - 落地 KnowledgeBase、Document、Chunk 三类产品模型，并接入现有用户隔离。
  - 建立文档状态机：`uploaded -> parsed -> chunked -> indexed -> failed`。
  - 引入 RAG-Pro 风格 chunk 管理：普通分块、父子分块、chunk 预览、chunk 进度。
  - 引入 vector store adapter：优先 Milvus Lite/Milvus，保留 Chroma/Memory 作为兼容 backend。
  - 引入 BGE-M3 dense+sparse 设计、RRF hybrid search、rerank、retrieval trace。
  - V1.5.1 文件解析结果进入新 KB/Document/Chunk 链路，不再只停留在上传附件能力。
- API 与前端产品化：
  - 新增 `/api/v1/kb` 知识库 CRUD、文档上传/列表/详情/parse/chunk/index。
  - 保留并重构 `/api/v1/rag/search`、`/api/v1/rag/answer`、`/api/v1/rag/health`、`/api/v1/rag/stats`，新增 `/api/v1/rag/test-qa`。
  - 新增 `KnowledgeBaseWorkbench`：知识库列表、文档列表、状态、chunk 预览、index 状态、test QA、trace 展示。
  - AskPage 先兼容新 RAG trace 字段；知识库选择和检索模式选择已在 KnowledgeBaseWorkbench 落地，后续再把 AskPage 入口做成主交互。
