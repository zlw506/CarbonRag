# CarbonRag V1.1.6F Memory Local/Cloud Compatibility Hotfix

## Summary
- 本轮只修后端 `memory` 存储兼容，不改前端功能，不扩业务面。
- 当前故障点已经定位清楚：`backend/app/session/service.py` 的 `_get_memory_service()` 在 PostgreSQL 运行时仍显式传 `sqlite_db_path=None` 给 `MemoryStore`，而 `backend/app/memory/store.py` 无条件执行 `Path(sqlite_db_path)`，导致 `GET /api/v1/sessions/{id}`、`POST /api/v1/sessions/{id}/ask`、`POST /api/v1/sessions/{id}/ask/stream` 统一 500。
- 本轮实现口径固定为：
  - 同一套 service / API / runtime 代码
  - 本地 `SQLite fallback`
  - 云端 `PostgreSQL runtime store`
  - 只在 `memory store/factory/config` 层分流
  - 禁止再把 `None` 直接传给任何 `Path(...)`

## Key Changes
- `memory` 存储层改成显式后端分流：
  - 在 `backend/app/memory/store.py` 引入 `build_memory_store(...)` / `get_memory_store()` 的统一工厂。
  - 工厂按以下优先级选择后端：
    1. 若设置 `MEMORY_BACKEND`，严格按它走：`sqlite | postgres`
    2. 若未设置且 `DATABASE_URL` 存在，则走 `postgres`
    3. 否则走 `sqlite`
  - `sqlite` 路径规则写死：
    - `sqlite_db_path` 为 `None` 时自动回退 `DEFAULT_SESSION_DB_PATH`
    - 只有在 SQLite 后端下才允许执行 `Path(...)`
  - `postgres` 后端规则写死：
    - 不依赖 `sqlite_db_path`
    - 不触碰 `Path(...)`
    - 只走 PostgreSQL 连接与运行时表

- `session/service.py` 修复：
  - `_get_memory_service()` 不再直接构造 `MemoryStore(database_url=..., sqlite_db_path=getattr(self.store, "db_path", None))`
  - 改为调用 memory factory，传入：
    - `database_url=getattr(self.store, "database_url", None)`
    - `sqlite_db_path=getattr(self.store, "db_path", None)` 仅作为可选值交给工厂
    - `memory_backend=get_settings().memory_backend`
  - 目标是：PostgreSQL 模式下根本不要求 session store 提供 SQLite 路径

- `memory/store.py` 修复范围：
  - 构造器和所有连接入口都改成对 `sqlite_db_path=None` 安全。
  - 建议实现为两种具体 store：
    - `SQLiteMemoryStore`
    - `PostgresMemoryStore`
  - `MemoryService` 继续只依赖统一 store 接口，不分两套业务逻辑。
  - 若为了最小热修复不想大改调用面，可以保留 `MemoryStore` 作为兼容入口，但其内部必须改成通过后端工厂路由，而不是自己直接 `Path(...)`。

- 配置口径补齐：
  - 在 `backend/app/core/config.py` 新增 `memory_backend: str | None = None`，读取环境变量 `MEMORY_BACKEND`。
  - `.env.example` 增加：
    - `MEMORY_BACKEND=sqlite`
  - `docs/deploy/VPS_BACKEND_DEPLOY.md` 的生产环境变量示例增加：
    - `MEMORY_BACKEND=postgres`
  - `docs/DEVELOPMENT_BOOTSTRAP.md` 同步写明：
    - 本地默认 `MEMORY_BACKEND=sqlite`
    - 云端生产 `DATABASE_URL + MEMORY_BACKEND=postgres`

- 测试与回归：
  - 新增：
    - `backend/tests/test_memory_store_sqlite.py`
    - `backend/tests/test_memory_store_postgres.py`
    - `backend/tests/test_session_memory_compat.py`
    - `backend/tests/test_session_ask_stream_compat.py`
  - 明确覆盖：
    - SQLite 模式下 memory store 正常初始化
    - PostgreSQL 模式下 memory store 正常初始化
    - `sqlite_db_path=None` 不再触发 `Path(None)`
    - `GET /api/v1/sessions/{id}` 正常
    - `POST /api/v1/sessions/{id}/ask` 正常
    - `POST /api/v1/sessions/{id}/ask/stream` 正常
  - PostgreSQL 测试不要求真实数据库；沿用当前 `test_postgres_runtime_mode.py` 的 fake runtime DB / monkeypatch 模式即可。

- 发布与分支：
  - 热修分支：`feature/v1.1.6f-memory-compat-hotfix`
  - 修复完成后：
    - 本地验证通过
    - 合并到 `main`
    - 打标签 `v1.1.6F`
  - 由于仓库当前把 `release/cloud-stable` 定义为实际云端部署线，这轮热修验收后应把 `release/cloud-stable` 快进到同一 hotfix 提交，再让 VPS pull 这条稳定发布线。

## Public APIs / Interfaces
- 业务 API 不改 shape。
- 新增环境变量契约：
  - `MEMORY_BACKEND=sqlite | postgres`
- 选择规则冻结为：
  - 显式 `MEMORY_BACKEND` 优先
  - 否则 `DATABASE_URL` 存在时默认 `postgres`
  - 否则默认 `sqlite`

## Test Plan
- 单元测试：
  - memory store SQLite 初始化
  - memory store PostgreSQL 初始化
  - `sqlite_db_path=None` 安全回退
- 路由兼容测试：
  - `GET /api/v1/sessions/{id}`
  - `POST /api/v1/sessions/{id}/ask`
  - `POST /api/v1/sessions/{id}/ask/stream`
  - SQLite / PostgreSQL 两种后端模式都覆盖
- 发布前验证：
  - 登录
  - 打开已有 session
  - 普通 ask
  - 流式 ask
- 本轮不要求：
  - 前端代码修改
  - Netlify 配置修改
  - 新业务功能

## Assumptions
- 本轮以最小热修复为目标，优先修复 `memory` 的后端选择和 `None -> Path(...)` 崩溃，不顺手改 session/knowledge/report 的其他架构。
- PostgreSQL 仍是云端生产唯一 runtime DB；SQLite 仍是本地开发默认口径。
- `MEMORY_BACKEND` 是本轮新增的显式控制位；若未设置则按现有 `DATABASE_URL` 自动推断，避免再次出现“本地能跑、云端崩溃”的隐式假设。
