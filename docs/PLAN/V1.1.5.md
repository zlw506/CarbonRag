# CarbonRag V1.1.5 Chat UX Slimming + Streaming + Thinking State

## Summary
- 本轮只做 V1.1.4 之后**还没做好的部分**，不重复重做流式基础和 memory foundation。
- 当前仓库已经有：
  - `POST /api/v1/sessions/{id}/ask/stream`
  - `thinking / answer / done` 的 SSE 基础
  - ask 页的流式状态机雏形
  - `memory_state` 与 summary/context source 的基础展示
- 所以 V1.1.5 的主工作收敛为 4 件事：
  - 聊天工作台**瘦身重排**
  - 把现有 streaming 变成**真正的默认聊天体验**
  - 把 thinking / streaming / done 做成**更成熟的消息态**
  - 落 `AGENT_COLLAB_POLICY.md`，把默认 1 子 agent 的协作纪律写死

## Key Changes
- 聊天布局重做，目标是“中间聊天区绝对主导”：
  - Ask 页从当前 `280px / 主区 / 340px` 三栏常驻布局，改为：
    - 左侧会话栏：默认展开宽度约 `220px`，折叠宽度约 `72px`
    - 中间消息区：占绝对主区，消息气泡最大宽度放宽到桌面端更合理范围
    - 右侧依据 / 系统状态：**默认不常驻展开**，改为右上角“依据 / 状态”按钮唤起 `Drawer`
  - 左侧新增明确的折叠按钮；右侧面板改为抽屉，不再长期占据横向空间。
  - `SystemInfoPanel` 从 Ask 页常驻侧栏移出，统一收进右侧抽屉的“系统状态”分区。

- 输入区改为底部悬浮输入：
  - Ask 页输入区不再作为普通内容卡片放在消息流下方。
  - 改为固定在主区底部的悬浮输入栏：
    - 文本输入框
    - 发送按钮
    - 附件按钮
    - 当前 scope 的轻量标签
    - 正在上传 / 正在生成的最小状态
  - 输入区上方只保留最必要状态；知识条目管理入口、复杂上下文说明、系统说明全部移出底部主输入视觉焦点。
  - 消息流底部需要为悬浮输入区预留安全内边距，避免最后一条消息被遮住。

- 消息流像聊天工具，而不是功能块：
  - `user` 消息固定靠右，`assistant` 固定靠左，`system` 改成弱化细条提示样式。
  - assistant 生成态细分为：
    - `pending`
    - `thinking`
    - `streaming`
    - `done`
    - `error`
  - 生成中的 assistant 消息始终先占位，再逐步填充正文；不再用整段刷新替代真实生成过程。
  - citations 保持默认折叠，但从“右侧整块信息面板”改成：
    - 消息卡片下方只显示轻量来源摘要与展开入口
    - 详细内容放右侧抽屉中查看
  - thinking 区仍可折叠，但默认更紧凑，只在真的收到 reasoning chunk 或用户主动展开时展示详细文本。

- 上下文 / token / summary 信息瘦身：
  - 现有顶部大块 `memory_state` 信息压缩成一个轻量上下文胶囊。
  - 默认只显示：
    - 一个上下文占用小指示（条形或环形，二选一，推荐条形）
    - 一个压缩状态短标签
    - 一句 context source 摘要，例如“最近 6 轮 + 会话摘要 + 3 条依据”
  - 点击后展开详细信息：
    - 当前上下文占用（估算）
    - 最近消息数
    - 是否已压缩
    - 摘要更新时间
    - 已摘要覆盖消息数
  - 这一块保留在消息区顶部，但视觉优先级降到明显次于消息流。

- Streaming / thinking 后端只做“补齐和冻结”，不重造：
  - 继续沿用现有 `POST /api/v1/sessions/{id}/ask/stream`，不新增第二套流式协议。
  - 对外 SSE 契约冻结为：
    - `message_start`
    - `status`
    - `thinking_delta`
    - `answer_delta`
    - `metadata`
    - `done`
    - `error`
  - 本轮后端只补缺口，不再重写 orchestrator：
    - 保证 `thinking -> delta -> done` 事件顺序稳定
    - 保证无 reasoning chunk 时仍有 `status=thinking`
    - 保证 `metadata` 一定带最终 `citations`、`source_summary`、`memory_state`
  - `thinking` 文本继续只做运行时 UI 数据，不落库。

- Memory 这轮只做体感修正，不改体系：
  - 保持现有 V1.1.3 的 `session_summary + recent_messages + memory_notes + grounding` 基础。
  - 本轮只把规则显性化并固定：
    - `session_summary`
    - 最近 6 轮高优先消息
    - grounding 结果
    - 当前问题
  - Ask 页用更轻量但明确的方式提示：
    - “已压缩旧消息”
    - “本轮使用最近 6 轮 + 会话摘要”
    - “本轮使用 N 条依据”
  - 不新增 memory 页面，不新增长期记忆写入 UI。

- 子 agent 协作纪律落地：
  - 新增 `docs/engineering/AGENT_COLLAB_POLICY.md`
  - 固定规则：
    - 默认只允许 `1` 个子 agent 协同
    - 只有你明确批准时，才允许临时升到 `2~3`
    - 升级时必须在文档/任务记录中留痕
    - 不允许默认多 agent 并发乱跑
  - 如果仓库没有现成 agent 配置文件，则本轮**只落文档**，不凭空引入配置系统。

- 文档新增 / 更新：
  - 新增：
    - `docs/research/claw-code/08_STREAMING_CHAT_UX_NOTES.md`
    - `docs/architecture/CHAT_STREAMING_FLOW.md`
    - `docs/architecture/CHAT_CONTEXT_COMPACTION_RULES.md`
    - `docs/engineering/AGENT_COLLAB_POLICY.md`
    - `docs/PLAN/V1.1.5.md`
  - 更新：
    - `README.md`
  - 文档口径只收敛到本轮主题，不再继续扩后台/知识流主线。

## Public APIs / Interfaces
- 不新增新的聊天业务 API。
- 继续沿用：
  - `POST /api/v1/sessions/{id}/ask`
  - `POST /api/v1/sessions/{id}/ask/stream`
- 本轮只冻结并文档化 `ask/stream` 的 SSE 事件契约，不再扩接口面。
- 前端行为改为：
  - Ask 页默认优先走 `/ask/stream`
  - 非流式 `/ask` 只作为后备降级路径

## Test Plan
- 前端 smoke：
  - 左侧会话栏明显缩窄，且可折叠
  - 右侧依据 / 系统状态不再常驻占宽，通过按钮打开 Drawer
  - 输入区始终固定在底部可见区域
  - 用户消息靠右、AI 消息靠左、system 消息弱化
  - 发送后用户消息立即出现，assistant 立即进入 `pending -> thinking -> streaming -> done/error`
  - thinking 有 heartbeat / pulse / three-dots 之类的明确动态表现
  - citations 默认折叠，不抢正文
  - context 胶囊默认轻量，点开后才显示详细上下文信息
  - 切换 session 时在途流式请求不会串到别的会话

- 后端回归：
  - `POST /api/v1/sessions/{id}/ask/stream` 继续通过现有 streaming 契约测试
  - provider 在“有 reasoning / 无 reasoning / provider error”三种情况下都能给出稳定事件序列
  - `metadata` 始终包含最终 citations、source_summary、memory_state
  - 非流式 `/ask` 兼容入口不退化

- 文档验收：
  - `08_STREAMING_CHAT_UX_NOTES.md` 明确写清本轮从 claw-code-study 和 `fix/ui-parity` 借了哪些点
  - `CHAT_STREAMING_FLOW.md` 写清前后端事件流
  - `CHAT_CONTEXT_COMPACTION_RULES.md` 写清 summary / 最近轮次 / grounding 的拼装与裁剪顺序
  - `AGENT_COLLAB_POLICY.md` 写死默认 1 子 agent、2~3 需明确批准的规则

## Assumptions
- 当前仓库已有的流式 ask foundation 属于已完成项，V1.1.5 不再重新发明，只做“默认体验化 + 契约冻结 + UI 收口”。
- `main` / `release/cloud-stable` 的主线纪律在现有文档里已经有基础，本轮不再把分支 housekeeping 作为主工作。
- 右侧依据 / 系统状态面板采用 `Drawer` 是本轮默认方案；移动端沿用同一路径，不再保留桌面端常驻右栏。
- 输入区固定悬浮优先用 CSS sticky/fixed + 主区底部留白实现，不引入新的布局框架或重写整个 AppShell。
- 本轮不扩后台、不扩知识流、不做新的 business module、不做完整长期 memory、不做插件系统。
