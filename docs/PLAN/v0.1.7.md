# CarbonRag v0.1.7 Conversation Experience + Session Foundation

## Summary
- 本轮从 v0.1.6 的“单轮 grounded ask”推进到“最小可用对话工作台”，核心不是扩业务，而是补三层基础：`session`、单会话上下文延续、文件上传入口骨架。
- 已拍板的关键实现口径：
  - 问答入口改为 `POST /api/v1/sessions/{id}/ask`
  - session 持久化底座使用 `SQLite`
  - 文件上传记录从第一天就绑定 `session_id`
- 当前仓库真实基线是：只有 `POST /api/v1/ask`、没有 `session/files` 模块、前端问答页仍是单页表单，`ai_runtime` 还没有真正消费会话历史。

## Key Changes
- 后端新增 `session` 模块，拆出 `schemas / store / service / adapters`，底层落到单文件 SQLite，推荐固定在 `data/outputs/runtime/carbonrag.sqlite3`，不新增新的 env 变量。
- session 数据模型至少固定为：`session_id`、`title`、`created_at`、`updated_at`、`messages[]`、`files[]`。消息项固定包含 `message_id`、`role`、`content`、`created_at`；助手消息额外持久化 `status`、`trace_id`、`citations`。
- 本轮公开 session API 固定为：
  - `POST /api/v1/sessions`
  - `GET /api/v1/sessions`
  - `GET /api/v1/sessions/{id}`
  - `POST /api/v1/sessions/{id}/ask`
  - `PATCH /api/v1/sessions/{id}` 只支持改 `title`
- `DELETE /api/v1/sessions/{id}` 本轮不做，避免把范围扩成完整会话治理。
- `POST /api/v1/ask` 在 v0.1.7 不再作为主入口；前端、文档、测试全部切到 session 路由，不保留双入口并行语义。
- ask 链路改为：先读取当前 session 最近消息，再做 `policy_retrieve`，再进 provider，最后把用户消息和助手消息一起回写 session。provider 失败也要落一条结构化助手错误消息，不能只让前端 toast。
- `context_builder` 正式支持 `session_context`，默认只带最近 4 轮完整对话，也就是最多 8 条历史消息；不做跨会话记忆，不做自动总结压缩，只预留后续扩展位。
- 新增 `files` 模块，公开 `POST /api/v1/files`，使用 multipart/form-data，字段固定为 `session_id` + `file`。文件落到 `data/outputs/uploads/{session_id}/`，返回 `file_id`、`session_id`、`filename`、`size`、`mime_type`、`stored_at`。
- 本轮上传文件不进入 retrieval、不参与 ask、不做解析，只作为 session 绑定资产和 UI 入口骨架。
- 前端问答页升级为 chat workspace：
  - 左侧 session 列表
  - 中间消息流
  - 底部输入区 + 发送按钮 + 附件按钮
  - 右侧 citations/source panel，移动端降级为折叠或 drawer
- session 列表至少支持新建、切换、高亮当前项，列表按 `updated_at` 倒序。
- 输入区选择文件后立即上传到当前 session，并显示已上传文件名/附件 chip；这些附件本轮只显示，不驱动回答。
- 助手消息下不再整块堆 citation，而是“消息 + 查看依据”联动 source panel；panel 展示标题、来源、片段、链接。
- 新会话标题默认 `新对话 YYYY-MM-DD HH:mm`；第一轮成功回答后，如果标题仍是默认值，则自动改为首个问题的截断文本。`PATCH` API 先落后端，前端本轮不强求手动重命名入口。
- 继续推进第三方学习线，新增 `docs/research/claw-code/06_SESSION_MEMORY_MAPPING.md`，只写 `session/state/memory/skills` 对 CarbonRag 的映射、已落地项和预留项。

## Public APIs / Types
- `POST /api/v1/sessions`：创建会话，返回 `SessionSummary`。
- `GET /api/v1/sessions`：返回会话列表，按最近更新时间倒序。
- `GET /api/v1/sessions/{id}`：返回 `SessionDetail`，包含 `messages[]` 与 `files[]`。
- `POST /api/v1/sessions/{id}/ask`：请求体沿用当前 ask 最小字段 `question / knowledge_scope / top_k`，不再带 `session_id`；响应体保持 `answer / mode / status / citations / trace_id`，并额外保证结果已落入该 session。
- `PATCH /api/v1/sessions/{id}`：只接受 `{ "title": "..." }`。
- `POST /api/v1/files`：multipart 上传，返回 `UploadedFile` 元信息。
- 前端内部类型新增 `SessionSummary`、`SessionDetail`、`SessionMessage`、`UploadedFile`；现有 `AskResponse` 保持不破，只是调用路径改成 session-scoped ask。
- `knowledge_scope` 继续只允许 `public`；`private_sample`、`mixed` 仍返回 `422`，因为本轮是 conversation/session foundation，不是 private grounding。

  - `test_session_service.py`：新建会话、自动标题、消息追加、最近历史窗口截取。
  - `test_ask_route_with_session.py`：`POST /sessions/{id}/ask` 会读取历史、写回用户/助手消息、保留 citations。
  - `test_file_upload_route.py`：multipart 上传成功、落盘成功、返回 file metadata、非法 `session_id` 报错。
- 后端更新：
  - 现有 ask 相关测试全部切到 session 路由。
  - `context_builder` 和 orchestrator 测试补 session history 断言。
- 前端 smoke：
  - 新建 session A，连续问两轮，消息流保留两轮历史。
  - 新建 session B，再问一轮，切回 A 时消息仍在。
  - 助手消息可打开 citations/source panel。
  - 上传一个文件后，输入区或会话附件区能看到文件名与状态。
  - provider 错误时，当前 session 仍留下用户消息和受控错误助手消息。
- 构建与运行：
  - 后端继续通过 pytest。
  - 前端继续通过 `typecheck` 和 `build`。
  - 文件上传所需依赖补 `python-multipart`，并纳入后端依赖与测试环境。

## Assumptions
- 本轮推荐新分支名为 `feature/conversation-session-foundation`。
- SQLite 与上传文件都放在 `data/outputs/` 下，沿用现有 ignore 规则，不新增配置项。
- 本轮只做单会话上下文，不做跨会话记忆、自动摘要、长期 memory。
- 本轮实现 `PATCH title` 后端接口，但不强制做前端改名 UI；`DELETE session` 延后。
- 文件上传采用保守文档类白名单和大小上限，默认按 20 MB 级别控制；上传成功只绑定 session，不参与 ask 推理。
