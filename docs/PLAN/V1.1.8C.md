# CarbonRag V1.1.8C Ask 页滚动容器收口

## Summary
- 本轮只修 Ask 聊天页的滚动归属，不动新业务、不改接口。
- 当前问题已经定位清楚：Ask 页虽然已有局部滚动容器，但 `AppShell`、`Content`、`chat-workbench`、`chat-workbench__main` 这几层没有把视口高度锁死；同时左侧栏用 `sticky`、中间主区用 `min-height`、消息流用 `max-height`，导致页面总高度仍然超过视口，于是浏览器出现“全局滚动条”。
- 本轮目标固定为：
  - 进入 Ask 页时，不出现页面级全局滚动
  - 左侧 session 堆叠自己滚
  - 中间聊天消息流自己滚
  - 底部 composer 固定可见
  - 右侧 drawer 继续独立滚，不影响主布局

## Key Changes
- Ask 路由进入“锁定视口布局”：
  - 在 `frontend/src/layouts/AppShell.tsx` 为 Ask 路由增加专用壳层态，例如 `app-shell--chat-locked`
  - Ask 路由下的 `Layout / Content` 改为基于 `100dvh` 的固定高度布局，不再依赖页面自然撑高
  - Ask 路由下关闭页面级 `overflow-y`，把滚动权交给内部容器
  - 非 Ask 页保持当前行为，不影响 Report / Calc / Knowledge / Admin

- 重定义 Ask 页滚动责任：
  - `frontend/src/pages/AskPage/index.tsx` 的根容器 `chat-workbench` 改成“占满可用高度”的容器，不再靠 `min-height` 撑页面
  - `chat-workbench__sidebar` 不再用当前这种会把整页高度撑开的 sticky 方案；改为固定高度容器
  - 左侧 session 区内部的列表容器单独 `overflow-y: auto`
  - `chat-workbench__main` 改成固定高度的纵向 flex 容器，`min-height: 0`
  - 中间消息卡片区域和 composer 之间的高度关系写死：
    - 上方 alert / context capsule / stream card header 占内容高度
    - 中间 `chat-message-stream` 吃剩余空间并独立滚动
    - 下方 composer 固定在主区底部，不再参与页面总高度计算
  - 右侧 `Drawer` 保持独立滚动，不参与页面主轴滚动

- CSS 实现口径锁死：
  - 只在 Ask 路由启用：
    - `height: 100dvh`
    - `overflow: hidden`
    - `min-height: 0`
  - 不再依赖：
    - `chat-workbench__main { min-height: calc(100vh - 132px) }`
    - `chat-message-stream { max-height: calc(100vh - 300px) }`
    这类“估算高度”写法
  - 改为父容器高度锁定 + 子容器 flex 分配，避免不同顶部内容高度下再次漏出全局滚动条
  - 消息流继续保留底部安全留白，避免被 composer 遮挡

- 组件结构上需要的小调整：
  - 左侧 `Card` 的 body 改成 `display: flex; flex-direction: column; min-height: 0`
  - session list 容器单独包一层滚动容器，而不是让整个 sidebar/card 跟着页面走
  - 中间 `chat-workbench__stream-card` 的 body 改成 `display: flex; flex-direction: column; min-height: 0`
  - `chat-message-stream` 改为 `flex: 1 1 auto; min-height: 0; overflow-y: auto`
  - composer 继续 sticky/floating，但锚定在主区底部，不再依赖整个页面滚动上下文

## Public APIs / Interfaces
- 不新增 API
- 不改：
  - `POST /api/v1/sessions/{id}/ask`
  - `POST /api/v1/sessions/{id}/ask/stream`
- 只改前端布局行为契约：
  - Ask 页不再出现页面级滚动条
  - 左侧会话列表与中间消息流分别成为独立滚动区

## Test Plan
- 桌面宽度下进入 Ask 页：
  - 浏览器页面本身无垂直滚动条
  - 左侧 session rail 内容过多时，仅左侧列表滚动
  - 中间聊天消息过多时，仅消息流滚动
  - composer 始终固定在底部可见区域
- 切换 session、发送消息、流式输出中：
  - 全局页面仍不出现滚动
  - 消息流自动滚到底部仍正常
  - thinking / streaming 状态不受影响
- 打开右侧依据 drawer：
  - drawer 自己滚动
  - 不重新触发页面级滚动
- 窄屏宽度下：
  - Ask 页仍不出现“三层一起滚”的情况
  - 左侧 rail 折叠/展开不破坏中间主区高度计算

## Assumptions
- 本轮只修 Ask 页；其他页面即使仍是普通页面滚动，也不在这轮处理范围内。
- 继续保留当前 focus mode 默认行为；本轮不重做 Ask 页视觉，只修滚动归属。
- 以 `100dvh` 为优先视口单位，避免移动端和桌面浏览器顶部 UI 变化带来的高度误差。
