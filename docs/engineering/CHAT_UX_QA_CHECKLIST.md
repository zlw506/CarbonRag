# Chat UX QA Checklist

本清单用于 V1.1.7、V1.1.8B 及后续产品化体验轮次，明确把“能用”和“感觉好”拆成两个不同验收门。

## 测试尺寸

- 桌面正常宽度
  - 建议：`1440px` 左右
- 更窄宽度
  - 建议：`1024px` 左右

## 截图要求

每次 UX pass 至少保留：

- `before` 首屏截图
- `after` 首屏截图
- `after` 发送一条消息后的生成中截图
- `after` 展开依据抽屉截图
- `after` 报告页可生成状态截图
- `after` 碳核算页结果截图
- `after` 我的知识库页截图

截图文件可先保存在本地评审目录，不要求纳入仓库。

## 第一层：能用

- 输入框始终易于发现
- 发送后用户消息立即进入消息流
- assistant 占位立即出现
- assistant 能稳定经历 `pending -> thinking -> streaming -> done/error`
- 右侧依据/状态抽屉能正常打开关闭
- 左侧 session rail 能正常折叠和切换 session
- focus mode 下不会影响正常发送、流式输出和会话切换
- 当前 session 可以直接生成报告，或得到明确下一步引导
- 普通用户默认看不到管理员入口
- 碳核算页和知识库页不会被系统信息面板长期占满侧边空间

## 第二层：感觉好

- 首屏视觉重心明显在聊天区，而不是在导航、顶部说明或依据面板
- composer 看起来是“随时可输入”的，而不是一个功能块
- thinking 和 streaming 的差异一眼可见
- 正文优先级明显高于 trace、反馈、context 等工程信息
- citations 可达但不抢正文注意力
- session rail 可快速扫描，标题比元数据更突出
- 管理员身份存在感不应破坏普通聊天产品感
- 报告页看起来像“可直接产出文档”的用户页面，而不是来源配置死路
- 碳核算页先给结论、再给明细，结果阅读顺序自然
- 我的知识库页更像用户内容页，而不是后台任务表格

## 逐项检查

### 1. 首屏产品感

- 是否一眼能看出哪里输入、哪里出答案、哪里看依据
- 是否仍有“三条侧栏一起抢注意力”的感觉

### 2. Composer

- 是否始终停留在底部可见区
- scope、附件、知识条目状态是否已经压缩成紧凑 chips
- 是否不需要为了继续对话而滚到页面底部

### 3. Streaming

- 是否能明显感到“AI 先思考，再生成”
- 是否避免了“等一会儿整段一下子出现”的感受

### 4. 可读性

- 正文段落是否足够松弛
- 标题、列表、段间距是否更容易读
- trace id 和诊断信息是否已经下沉

### 5. 依据与上下文

- 依据面板是否只在需要时才抢占空间
- context 指示是否默认轻量
- 用户是否能理解“本轮用了最近消息 + 会话摘要 + 依据”

### 6. 报告页

- 进入当前 session 后是否能直接生成可用报告
- 缺少来源时是否给出明确引导，而不是只剩禁用按钮
- 报告正文是否优先于来源规则和工程说明

### 7. 碳核算页

- 字段名和帮助文本是否更接近用户语言
- 结果区是否先显示结论，再按需展开分项和因子
- 是否能自然理解结果和当前会话的关系

### 8. 我的知识库

- 是否一眼能看出“我有哪些内容”
- 状态、可操作项是否清楚
- ingest / rebuild / retry 等后台指标是否已退后

### 9. 管理员入口与角色污染

- 普通用户是否在主路径上看不到后台入口
- 管理员身份标识是否被弱化到不影响普通聊天首屏
- 权限边界是否通过 UI 和路由都保持一致
