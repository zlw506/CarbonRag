# Agent Collaboration Policy

本文件冻结开发期的子 agent 协作纪律。

## 默认规则

- 默认只允许 `1` 个子 agent 协同
- 不允许默认开 `2~3` 个子 agent 并发乱跑
- 主线程必须保留集成、接口对齐和最终回归责任

## 什么时候允许升到 2~3 个

只有在以下前提同时满足时才允许：

1. 任务确实可拆成清晰的独立子问题
2. 写集边界可明确分开，避免互相覆盖
3. 你本人明确批准临时提升到 `2~3` 个子 agent

## 典型适用场景

### 默认 1 个子 agent

适合：
- 只读研究
- 文档映射沉淀
- 独立测试补充
- 非关键路径的局部探查

### 临时 2~3 个子 agent

只适合：
- 前后端明确分离的并行开发
- 一个主线程 + 一个文档/测试线程 + 一个独立模块线程
- 写集彼此不重叠的情况

## 明确禁止

- 没有批准时默认多 agent 并行
- 多个子 agent 同时修改同一批文件
- 把主线程的关键集成工作全部外包给子 agent
- 为了“看起来快”而制造额外并发噪声

## 留痕要求

当使用子 agent 时，必须在任务记录、计划或交付说明里写清：

- 本轮用了几个子 agent
- 每个子 agent 负责什么
- 哪些文件由主线程集成

## 当前仓库口径

V1.1.5 起，CarbonRag 的默认开发纪律固定为：

- 默认 `1` 个子 agent
- `2~3` 个只在明确批准后临时启用

如果未来新增本地 agent 配置文件或仓库级 agent 配置系统，应继续遵守这条默认规则，而不是反过来放宽默认并发。

## V1.2.1 OpenSpec / PR 协作补充

- 多 agent 或多人协作的任务必须先声明影响模块和 OpenSpec change-id。
- 子 agent 只允许在清晰写集边界内工作，不能绕过 PR 模板、CODEOWNERS 或 `Git-ys1` 最终审查。
- 只读研究、spec-gen 草稿、OpenSpec 草案必须和正式 specs 区分。
- 任何进入 `main` 的改动都要能追溯到 PR、验证结果和模块 owner 审查。

## V1.1.7 轮次补充

V1.1.7 是聊天体验审计轮，纪律进一步收紧：

- 本轮默认只允许 `1` 个子 agent
- 即使会话环境中存在历史子 agent，也不把它们视为本轮可默认并发使用
- 本轮若确需调用子 agent，只允许只读研究或测试侧探查
- 本轮不允许多写者并发修改聊天主界面文件
