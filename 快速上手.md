## 新人最短路径

### #2 / #3 从云端开始

```powershell
git clone https://github.com/<your-github-username>/CarbonRag.git
cd CarbonRag
git remote add upstream https://github.com/Git-ys1/CarbonRag.git
git fetch upstream
git switch -c t2/v1.2/onboarding-smoke upstream/main
```

### 校验 OpenSpec

```powershell
openspec list
openspec validate --all
```

### 初始化本地环境

```powershell
powershell -NoProfile -ExecutionPolicy Bypass -File scripts/bootstrap.ps1
```

### 启动本地前后端

```powershell
powershell -NoProfile -ExecutionPolicy Bypass -File scripts/dev-local.ps1
```

访问：

- 前端：`http://127.0.0.1:5173`
- 后端：`http://127.0.0.1:8000`
- 健康检查：`http://127.0.0.1:8000/healthz`

## 每轮开发怎么开始？

1. 先同步 `main`。
2. 建自己的分支。
3. 跑 `openspec validate --all`。
4. 新功能先创建 `openspec/changes/<change-id>/`。
5. 让 Codex 读取 `AGENTS.md`、`openspec/AGENTS.md`、`openspec/specs/**` 和本轮 change。
6. 实现后跑测试。
7. 提 PR，填写模板。

详细流程看：[docs/governance/OPENSPEC_CODEX_WORKFLOW_RUNBOOK.md](docs/governance/OPENSPEC_CODEX_WORKFLOW_RUNBOOK.md)

## 哪些文件别人拿不到也正常？

这些不是“原版缺失”，而是每台机器自己生成：

- `.env`
- `frontend/.env.local`
- `node_modules/`
- Python `.venv` / `backend/.conda`
- SQLite / runtime data
- uploads
- `.spec-gen/`
- `3rdparty/spec-gen/`

恢复方式看：[docs/governance/TRACKED_COLLABORATION_ASSET_INVENTORY.md](docs/governance/TRACKED_COLLABORATION_ASSET_INVENTORY.md)

## #1 审查别人代码

```powershell
gh pr checkout <PR编号>
git fetch origin main
git diff origin/main...HEAD
openspec validate --all
```

然后让 Codex 只读审查 diff，最后由 #1 人工 approve / comment / request changes。

详细流程看：[docs/governance/PR_REVIEW_RUNBOOK.md](docs/governance/PR_REVIEW_RUNBOOK.md)

## 当前最重要的规则

- 首席团队重大决策先看 [开发公告.md](开发公告.md)。
- `main` 是稳定源码基线，也是所有席位共同入口。
- #2/#3 必须 fork-and-PR。
- `Git-ys1` 是 `main` 最终管理员和 PR 最终审查人。
- 新功能没有 OpenSpec change-id，不准直接施工。
- 忽略文件不要传云端，协作资产必须传云端。
