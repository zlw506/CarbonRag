## 新人最短路径

### #2 / #3 从云端开始

先在 GitHub 网页 fork 主仓库：

```text
https://github.com/Git-ys1/CarbonRag
```

然后 clone 自己的 fork，并把首席仓库设为 `upstream`：

```powershell
git clone https://github.com/<your-github-username>/CarbonRag.git
cd CarbonRag
git remote add upstream https://github.com/Git-ys1/CarbonRag.git
git fetch upstream
git switch -c t2/v1.2/onboarding-smoke upstream/main
```

从这里开始，#2/#3 的本地仓库应该有两个 remote：

```powershell
git remote -v
```

- `origin`：自己的 fork，用来 push 自己的分支。
- `upstream`：`Git-ys1/CarbonRag`，只用来同步主线。

### 校验 OpenSpec

```powershell
openspec list
openspec validate --all
```

### 初始化本地环境

```powershell
powershell -NoProfile -ExecutionPolicy Bypass -File scripts/bootstrap.ps1
```

### 启动本地前后端

```powershell
powershell -NoProfile -ExecutionPolicy Bypass -File scripts/dev-local.ps1
```

访问：

- 前端：`http://127.0.0.1:5173`
- 后端：`http://127.0.0.1:8000`
- 健康检查：`http://127.0.0.1:8000/healthz`

### #2 / #3 第一次提交 PR

新席位不要直接 push `main`。完成修改和验证后，推送自己的分支到 fork：

```powershell
git status -sb
openspec validate --all
git push -u origin t2/v1.2/<topic>
```

如果安装了 GitHub CLI，可以直接创建 PR：

```powershell
gh pr create --repo Git-ys1/CarbonRag --base main --head <your-github-username>:t2/v1.2/<topic>
```

如果没有 `gh`，就在 GitHub 网页打开自己的 fork，点 `Compare & pull request`，目标固定为：

```text
Git-ys1/CarbonRag:main
```

PR 必须填写 `.github/PULL_REQUEST_TEMPLATE.md`，并声明 OpenSpec change-id、影响模块、风险、验证结果和审批状态。

## 长期开发主流程：propose -> apply -> archive

先纠错：下面这些都不是 PowerShell 命令，不要在终端里敲：

```powershell
openspec propose
openspec-propose
/opsx:propose
```

`propose / apply / archive` 是工作流阶段名，不是当前 OpenSpec CLI 的顶层命令。PowerShell 里真实可执行的是 `openspec new change`、`openspec status`、`openspec instructions`、`openspec validate`、`openspec archive`。

真正开工不是看 `openspec show governance`。`governance` 只是协作规则。真正开工要创建一个具体 change，然后走三步：

1. `propose`：先生成提案、设计、任务和增量规格，不写代码。
2. `apply`：#1 审查通过后，Codex 按 tasks 实现。
3. `archive`：验证通过后，把增量规格合并进主规格。

### 最省事的开工方式

你不需要每轮亲自操作 OpenSpec。以后正常给 Codex 发任务书即可，格式越简单越好：

```text
执行 Vx.y.z。
目标：<一句话目标>。
范围：<做什么 / 不做什么>。
请按 CarbonRag OpenSpec 工作流开工。
```

Codex 必须自动完成这些 housekeeping：

1. 派生或确认 `change-id`。
2. 读取 `AGENTS.md`、`openspec/specs/**`、`docs/governance/**`。
3. 创建或检查 `openspec/changes/<change-id>/`。
4. 生成或检查 `proposal.md`、`design.md`、`tasks.md`、delta spec。
5. 运行 `openspec validate --all`。
6. 停在 propose 阶段，等 #1 审查；除非你明确授权快进执行。

如果你想手动起一个 change，只运行这个脚本即可：

```powershell
powershell -NoProfile -ExecutionPolicy Bypass -File scripts/openspec-start-change.ps1 -Id <change-id> -Goal "<一句话目标>" -Domain <domain>
```

脚本会创建 change、显示状态，并打印一段可以直接发给 Codex 的标准提示词。

### #2 / #3 正式开工必须基于 OpenSpec

#2/#3 拿到任务后，不要直接让 AI 改代码。先让 Codex 或脚本创建 change：

```powershell
powershell -NoProfile -ExecutionPolicy Bypass -File scripts/openspec-start-change.ps1 -Id <change-id> -Goal "<一句话目标>" -Domain <domain>
```

然后把脚本输出的提示词发给 Codex。Codex 必须先完成 propose 阶段，等 #1/负责人审查后，才进入 apply 阶段。

一句话规则：

```text
没有 openspec/changes/<change-id>/，不准开始实现新功能。
```

### PowerShell 里实际敲什么？

```powershell
openspec list
openspec list --specs
openspec validate --all
openspec new change <change-id>
openspec status --change <change-id>
openspec instructions <artifact-id> --change <change-id>
openspec validate <change-id> --strict
```

说明：

- `<change-id>` 用本轮任务名，例如 `add-report-export`。
- `<artifact-id>` 以 `openspec status --change <change-id>` 的输出为准，常见是 `proposal`、`design`、`tasks`。
- 如果当前 Codex 客户端能识别 `.codex/skills/openspec-propose`，它会替你执行这些真实命令；如果不能识别，就按上面的 PowerShell 命令手动跑。

归档时才运行：

```powershell
openspec archive <change-id> --yes
openspec validate --all
```

### 第一步：propose 阶段

把下面这段发给 Codex，不是发给 PowerShell：

```text
执行本轮 OpenSpec propose 阶段。
你可以处理 Git 和 OpenSpec，但必须遵守 AGENTS.md、openspec/AGENTS.md、openspec/specs/**、docs/governance/**。
先检查当前工作区、当前分支、远端状态、OpenSpec 状态和未忽略文件。
不要重复 openspec init。
如需运行 openspec update .，遇到删除 openspec/AGENTS.md 或 instruction 文件的提示，默认选择 n。
先输出你准备执行的 Git/OpenSpec 步骤；只有安全的只读检查可以直接执行。
涉及 commit、push、merge、rebase、删除、reset 的操作，必须说明原因和目标后再执行。
本轮 change-id: <change-id>。
本轮目标: <一句话目标>。
只做 propose：生成或检查 proposal.md、design.md、tasks.md、specs/<domain>/spec.md。
生成后停止，等待我审查，不要 apply。
```

### 第二步：apply 阶段

#1 审查 proposal/design/tasks/spec delta 通过后，再发：

```text
执行本轮 OpenSpec apply 阶段。
change-id: <change-id>。
先读取 openspec/changes/<change-id>/** 和相关 openspec/specs/**。
严格按 tasks.md 实现。
不要擅自扩大范围。
涉及 commit、push、merge、rebase、删除、reset 的操作，必须说明原因和目标后再执行。
实现后运行本轮要求的测试，并汇总结果。
```

注意：当前 OpenSpec CLI 没有 `openspec apply` 顶层命令。`apply` 阶段就是让 Codex 读取 `openspec/changes/<change-id>/**`，按 `tasks.md` 实现代码。

### 第三步：archive 阶段

实现验证通过后，再发：

```text
执行本轮 OpenSpec archive 阶段。
change-id: <change-id>。
先确认 tasks.md 全部完成、测试通过、文档已同步。
然后归档 change，把增量规格合并进 openspec/specs/**。
归档后运行 openspec validate --all。
```

归档阶段对应真实 PowerShell 命令：

```powershell
openspec archive <change-id> --yes
openspec validate --all
```

固定原则：

1. 当前仓库已经初始化过 OpenSpec，不再跑 `openspec init`。
2. 首次刷新用 `openspec update .`，但清理 `openspec/AGENTS.md` 默认选 `n`。
3. 新功能先有 `openspec/changes/<change-id>/`。
4. Codex 可以处理 Git，但必须按文档说明意图，不能自由 pull/merge/push。
5. proposal/design/tasks/delta spec 通过审查后才 apply。
6. `openspec show governance` 只是看协作规则，不是看当前任务。

详细流程看：[docs/governance/OPENSPEC_CODEX_WORKFLOW_RUNBOOK.md](docs/governance/OPENSPEC_CODEX_WORKFLOW_RUNBOOK.md)

## 哪些文件别人拿不到也正常？

这些不是“原版缺失”，而是每台机器自己生成：

- `.env`
- `frontend/.env.local`
- `node_modules/`
- Python `.venv` / `backend/.conda`
- SQLite / runtime data
- uploads
- `.spec-gen/`
- `3rdparty/spec-gen/`

恢复方式看：[docs/governance/TRACKED_COLLABORATION_ASSET_INVENTORY.md](docs/governance/TRACKED_COLLABORATION_ASSET_INVENTORY.md)

## #1 审查别人代码

```powershell
gh pr checkout <PR编号>
git fetch origin main
git diff origin/main...HEAD
openspec validate --all
```

然后让 Codex 只读审查 diff，最后由 #1 人工 approve / comment / request changes。

详细流程看：[docs/governance/PR_REVIEW_RUNBOOK.md](docs/governance/PR_REVIEW_RUNBOOK.md)

## 当前最重要的规则

- 首席团队重大决策先看 [开发公告.md](开发公告.md)。
- `main` 是稳定源码基线，也是所有席位共同入口。
- #2/#3 必须 fork-and-PR。
- `Git-ys1` 是 `main` 最终管理员和 PR 最终审查人。
- 新功能没有 OpenSpec change-id，不准直接施工。
- 忽略文件不要传云端，协作资产必须传云端。
