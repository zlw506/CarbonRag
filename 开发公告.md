# CarbonRag 开发公告

版本：V1.2.5 起启用

本文件是首席开发团队发布给所有席位的“认知层公告板”。它不替代 OpenSpec、PR 模板或任务书；它用于让所有协作者第一时间理解：项目方向、重大决策、关键变量、协作纪律发生了什么变化。

## 公告效力

- 发布方：#1 首席开发团队。
- 默认读者：#2、#3 及后续所有开发席位。
- 发布位置：必须合入并推送到 `main`。
- 权限口径：其他团队默认只读，按公告要求执行；如需质疑或补充，通过 PR 评论、Issue 或会议反馈，不直接改公告。
- 生效方式：公告进入 `main` 后生效。

## 什么时候必须发公告？

出现以下情况必须更新本文件：

- 分支、发布线、PR 审查规则变化。
- OpenSpec / Codex / GitHub CLI / VS Code 工作流变化。
- 模块边界、owner、权限边界变化。
- 环境变量、部署入口、本地测试入口变化。
- 数据安全、忽略规则、可提交资产范围变化。
- 重大技术路线、版本节奏或团队分工变化。
- 一项决定会影响 #2/#3 如何开工、测试、提交或审查。

## 公告格式

每条公告固定包含：

- 日期
- 版本
- 决策摘要
- 影响范围
- 所有席位必须执行的动作
- 关联文档

## 最新公告

### 2026-05-06 / V1.2.5 / 统一开放协作基线生效

#### 决策摘要

从 V1.2.5 起，CarbonRag 进入多席位协作准备阶段。`main` 被定义为所有席位共同协作入口、稳定源码基线和当前公网默认发布基线。#1 首席开发团队负责发布协作规则、审查 PR、维护 OpenSpec 规格库和公告板。

#### 影响范围

- Git 分支与 PR 流程
- OpenSpec + Codex 工作方式
- 本地测试与部署验证流程
- 可提交/必须忽略文件范围
- #2/#3 入场方式

#### 所有席位必须执行

1. 新席位先看 [快速上手.md](快速上手.md)。
2. 不重复执行 `openspec init`；CarbonRag 已经是完成初始化的 brownfield 项目。
3. 当前工作区开工时，让 Codex 按 `propose -> apply -> archive` 三个阶段执行，不要直接写代码。注意这三个是阶段名，不是 PowerShell 顶层命令。
4. 如运行 `openspec update .` 时提示删除 `openspec/AGENTS.md`，默认选 `n`。
5. 从 `main` fork/clone，不从 #1 本机目录复制。
6. 新功能必须先有 `openspec/changes/<change-id>`。
7. 每轮开工前至少确认：

```powershell
openspec list
openspec validate --all
```

8. #2/#3 必须 fork-and-PR，不得直接 push `main`。
9. PR 必须填写 OpenSpec、模块、风险、验证和批准字段。
10. 忽略文件不要上传；协作资产必须上传。

#### 关键变量

- `main`：稳定源码基线、共同入口、当前公网默认发布基线。
- `t1/v1.2/<topic>`：#1 开发分支。
- `t2/v1.2/<topic>`：#2/#3 fork 内开发分支。
- `Git-ys1`：`main` 最终管理员和 PR 最终审查人。
- `openspec/specs/**`：当前系统规格事实源。
- `openspec/changes/**`：拟议变更与任务流。

## 首次 OpenSpec 开发测试

下面是一轮最小、可回滚、低风险的 OpenSpec 开发测试。它只改文档，不碰业务代码。

### 1. propose 阶段

先在 PowerShell 里用真实 CLI 创建 change：

```powershell
openspec new change <change-id>
openspec status --change <change-id>
```

```text
执行本轮 OpenSpec propose 阶段。
你可以处理 Git 和 OpenSpec，但必须遵守 AGENTS.md、openspec/AGENTS.md、openspec/specs/**、docs/governance/**。
先检查当前工作区、当前分支、远端状态、OpenSpec 状态和未忽略文件。
不要重复 openspec init。
如需运行 openspec update .，遇到删除 openspec/AGENTS.md 或 instruction 文件的提示，默认选择 n。
先输出你准备执行的 Git/OpenSpec 步骤；只有安全的只读检查可以直接执行。
涉及 commit、push、merge、rebase、删除、reset 的操作，必须说明原因和目标后再执行。
本轮 change-id: openspec-smoke-test。
本轮目标: 验证 OpenSpec 工作流，不改业务功能。
只做 propose：生成或检查 proposal.md、design.md、tasks.md、specs/governance/spec.md。
生成后停止，等待我审查，不要 apply。
```

### 2. Codex 应先确认 OpenSpec 状态

```powershell
openspec update .
openspec list
openspec validate --all
```

如果 `openspec update .` 要删除 `openspec/AGENTS.md`，选 `n`。

### 3. 创建测试 change

```powershell
New-Item -ItemType Directory -Force openspec/changes/openspec-smoke-test/specs/governance
```

创建这些文件：

```text
openspec/changes/openspec-smoke-test/proposal.md
openspec/changes/openspec-smoke-test/design.md
openspec/changes/openspec-smoke-test/tasks.md
openspec/changes/openspec-smoke-test/specs/governance/spec.md
```

### 4. 让 Codex 先读规范再动手

给 Codex 的输入：

```text
先读取 AGENTS.md、openspec/AGENTS.md、openspec/specs/**、openspec/changes/openspec-smoke-test/**、docs/governance/**。
本轮 change-id: openspec-smoke-test。
本轮只做 docs-only 的 OpenSpec 工作流测试，不改业务代码。
请先检查 proposal/design/tasks 是否完整，再给出要改的文档清单。
```

### 5. validate change

```powershell
openspec validate openspec-smoke-test --strict
openspec validate --all
```

### 6. 提交测试分支

```powershell
git add .
git commit -m "v1.2.6: test openspec collaboration workflow"
git push -u origin t1/v1.2/openspec-smoke-test
```

### 7. apply 与 archive

#1 审查通过后，才让 Codex 执行 apply。`apply` 不是 `openspec apply` 命令，而是让 Codex 按 `tasks.md` 实现代码。验证通过后，再执行真实 CLI：

```powershell
openspec archive <change-id> --yes
openspec validate --all
```

审查和 PR 按 [docs/governance/PR_REVIEW_RUNBOOK.md](docs/governance/PR_REVIEW_RUNBOOK.md) 执行。
