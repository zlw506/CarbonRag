# CarbonRag 开发公告

版本：V1.2.5 起启用

本文件是首席开发团队发布给所有席位的“认知层公告板”。它不替代 OpenSpec、PR 模板或任务书；它用于让所有协作者第一时间理解：项目方向、重大决策、关键变量、协作纪律发生了什么变化。

## 公告效力

- 发布方：#1 首席开发团队。
- 默认读者：#2、#3 及后续所有开发席位。
- 发布位置：必须合入并推送到 `main`。
- 权限口径：其他团队默认只读，按公告要求执行；如需质疑或补充，通过 PR 评论、Issue 或会议反馈，不直接改公告。
- 生效方式：公告进入 `main` 后生效。

## 什么时候必须发公告？

出现以下情况必须更新本文件：

- 分支、发布线、PR 审查规则变化。
- OpenSpec / Codex / GitHub CLI / VS Code 工作流变化。
- 模块边界、owner、权限边界变化。
- 环境变量、部署入口、本地测试入口变化。
- 数据安全、忽略规则、可提交资产范围变化。
- 重大技术路线、版本节奏或团队分工变化。
- 一项决定会影响 #2/#3 如何开工、测试、提交或审查。

## 开发日志制度

从 V1.4.5 起，每一轮完成并准备发布云端前，负责席位必须写一份简短开发日志，和代码一起进入 `main`。

固定目录结构：

```text
日志/
├─ #1/
│  └─ V1.4.5/
│     └─ 开发日志.md
├─ #2/
└─ #3/
```

日志至少写清：

- 本轮版本号与负责席位
- 本轮目标
- 实际完成内容
- 验证结果
- 已知限制
- 后续交接事项

日志不是替代 OpenSpec，也不是替代 PR 描述；它是给项目负责人和其他席位快速理解“这一轮实际发生了什么”的可追溯记录。

## 公告格式

- 版本

### 2026-05-10 / V1.6.8 / Windows RAG-Pro 运行时切到 Docker Milvus Standalone

#### 决策摘要

V1.6.8 修正 RAG-Pro 运行时路线：native Windows 不再以 `milvus_lite + .db` 作为默认验收路径，Windows 开发者统一使用 Docker Milvus Standalone：

```env
RAG_VECTOR_BACKEND=milvus
RAG_MILVUS_URI=http://127.0.0.1:19530
RAG_EMBEDDING_PROVIDER=bge_m3
RAG_MODEL_AUTO_DOWNLOAD=false
RAG_REQUIRE_REAL_VECTOR=true
```

WSL/Linux/macOS 才继续使用 `RAG_VECTOR_BACKEND=milvus_lite` 与本地 `.db` 文件。`memory` 只允许开发调试，不能作为比赛验收。

RAG 开发阶段仍然准许完整迁移/改写 RAG-Pro 的核心结构与逻辑；但不得提交 `3rdparty` 源码树，且必须保留 CarbonRag 现有 auth/session/AskPage/report/OpenSpec/GitNexus/Mattermost 边界。

#### 影响范围

- 影响 RAG-Pro 向量库运行 profile、RAG trace、health/stats、smoke 脚本和团队部署说明。
- 不改变 #3 碳因子页，不接 ragPdfSystem 的 Celery/RabbitMQ/MinIO，不做知识图谱 UI。
- 2C2G VPS 不跑 Milvus；后续云端 Milvus 需要 4C/8GB 以上或独立向量服务。

#### 所有席位必须执行

Windows 本机：

```powershell
powershell -NoProfile -ExecutionPolicy Bypass -File scripts/rag-start-milvus-docker-windows.ps1
powershell -NoProfile -ExecutionPolicy Bypass -File scripts/rag-smoke-milvus-standalone.ps1
```

离线模型包仍放在：

```text
data/outputs/models/BAAI/bge-m3
data/outputs/models/BAAI/bge-reranker-v2-m3
```

Docker 镜像和 volume 可能占用 Docker Desktop 数据盘。C 盘空间不足时，必须先迁移 Docker Desktop / WSL 数据目录到大盘，再拉 Milvus 镜像。

### 2026-05-09 / V1.5.4 / AskPage 附件交互改为 ChatGPT 式草稿附件

#### 决策摘要

V1.5.3 的“会话文件”卡片交互不合格：它把附件放成了一个孤立中间卡片，还要求用户再次点选才加入本轮提问，违背主流 AI 聊天工具的附件心智。

V1.5.4 已改为 ChatGPT 式附件交互：

- 上传文件后，附件直接进入输入框下方的草稿附件区。
- 用户点 `X` 后，该附件会从本轮草稿中移除，不再反复冒回。
- 发送消息时，只把已解析完成的草稿附件传给后端。
- 发送后，附件跟随用户消息展示在消息流里，而不是常驻底边栏。
- 历史消息如果有 `private_upload` citation，会把对应文件反推显示到上一条用户消息附件区。

#### 影响范围

- 仅影响 AskPage 附件前端交互与样式。
- 不改变后端文件解析、`/api/v1/files`、`attached_file_ids` 字段名、#2 LightRAG、#3 碳因子页。

#### 所有席位必须执行

涉及聊天页、附件上传、文件读取体验的后续改动，必须以 V1.5.4 的交互为基线：**附件是消息草稿的一部分，发送后成为用户消息的一部分，不再做孤立文件卡片。**

#### 参考依据

- assistant-ui 的附件设计明确区分 `ComposerAttachments` 和 `UserMessageAttachments`。
- OpenAI 文件上传说明的产品心智是“在对话中上传文件并围绕文件提问”。

同时，GitNexus 不再只是“装好了备用”的工具。#1 在 V1.5.1 文件读取链路和 AskPage 热修中实测确认：GitNexus 对大型代码仓的结构定位、调用关系、影响面判断、PR 审查非常有价值。各席位必须开始常用 GitNexus，而不是继续只靠 `rg` 和手工猜测。

#### 当前 PR 状态

- #3 PR #8：`V1.4.8 #3 碳因子数据库基础版` 已收到，进入 #1 审查队列。
- #3 本轮方向仍属 V1.4.x 碳因子线；但合并前必须说明与当前 V1.5.1 文件上传基线、AskPage、运行时 schema 的关系。
- #2/#3 后续如果继续基于旧 `main` 扩大改动，冲突成本由对应席位自行消化。

#### GitNexus 使用经验

#1 这轮实际使用经验如下：

1. `gitnexus query "<概念>"` 比盲目 `rg` 更适合快速找代码入口，尤其适合查 `AskPage`、session、RAG、knowledge、文件上传、citation 这种跨模块链路。
2. `gitnexus impact <symbol>` 可以快速判断一个函数/页面上游影响面，适合改前评估爆炸半径。
3. `gitnexus detect_changes` 适合提交前看本轮实际影响了哪些 symbols 和 execution flows，PR 审查时必须作为重要证据。
4. 如果 `gitnexus status` 显示 stale，可以先用旧索引辅助理解历史结构，但不能把它当最终影响分析；大改、审 PR、合并前必须重建索引。
5. GitNexus 输出 HIGH / CRITICAL 不一定代表不能改，但代表 PR 描述和 Mattermost 汇报必须说明为什么风险可接受、跑了哪些测试。
6. GitNexus 是“结构导航 + 影响分析”工具，不替代 OpenSpec；OpenSpec 管需求边界，GitNexus 管代码结构，Codex 才负责执行。

#### 所有席位必须执行

1. 开工前固定执行：

```powershell
openspec validate --all
gitnexus status
```

2. 如果索引过旧，执行：

```powershell
gitnexus analyze --force --embeddings --skills --verbose --worker-timeout 60
```

3. 进入代码前先查结构：

```powershell
gitnexus query "你要改的功能或模块关键词"
gitnexus impact <函数名或文件符号> --direction upstream --depth 2
```

4. 提交或发 PR 前必须跑：

```powershell

```

5. Mattermost 的 `PLAN` / `CHANGED` / `REVIEW_READY` 中必须补一行 GitNexus 摘要：

```text
GitNexus：status=up-to-date/stale；impact=LOW/MEDIUM/HIGH/CRITICAL；detect_changes=已跑/未跑及原因。
```

6. #2/#3 Codex 必须把下面这条写入自己的本地记忆、AGENTS 或等效长期规则：

```text
CarbonRag 大项目开发中，非平凡代码修改前必须优先使用 GitNexus 查询结构、调用关系和影响范围；不要只靠 grep/rg 盲改。若 GitNexus stale，先说明并尽量重建索引。
```

### 2026-05-08 / V1.4.7C / Mattermost watcher 第二层唤醒增强

#### 决策摘要

#1 已确认：当前 Codex 没有稳定公开接口允许外部脚本把消息直接注入“正在打开的这个聊天输入框”。因此 watcher 不承诺做到真正无人值守唤醒当前 UI。

本轮改为更可靠的两层唤醒：

1. **本地交接文件**：任何 `PLAN` / `BLOCK` / `REVIEW_READY` 触发都会写入 `logs/coordination/latest-trigger.md`，方便 #1 人类或 Codex 接管。
2. **可选 Codex 启动**：watcher 可选打开 `codex resume --last`，也可在 `REVIEW_READY` 时启动一次只读 `codex exec` 审查草稿。

这些本地日志和触发文件不进入仓库。

#### 所有席位必须执行

1. #2/#3 继续按结构化格式发 `PLAN` / `CHANGED` / `REVIEW_READY`。
2. #1 watcher 只用于提醒和生成审查草稿，不自动 approve、merge、push。
3. 如果 #2/#3 继续 push 已批准 PR 的新 commit，必须重新发 `REVIEW_READY`。
4. #1 看到 watcher 触发后，仍要在当前 Codex 会话中明确下令审查或合并。

#### #1 推荐运行方式

```powershell
powershell -NoProfile -ExecutionPolicy Bypass -File scripts/coordination/watch-mattermost.ps1 `
  -LaunchCodexResume
```

如需对 `REVIEW_READY` 自动生成只读审查草稿：

```powershell
powershell -NoProfile -ExecutionPolicy Bypass -File scripts/coordination/watch-mattermost.ps1 `
  -LaunchCodexExecReview
```

重要边界：

- watcher 不能可靠地把消息直接塞进已经打开的 Codex UI 输入框。
- watcher 可以提示 #1，也可以打开一个恢复最近 Codex session 的 PowerShell 窗口。
- watcher 不允许自动 approve、merge、push 或修改业务代码。
- #1 人类仍是最终审查和合并决策者。

#### 所有席位必须执行

1. #2/#3 继续通过 Mattermost 发 `PLAN`、`CHANGED`、`REVIEW_READY`，不要等 PR 阶段才暴露风险。
2. #1 watcher 默认忽略 #1 自己的消息，只响应 #2/#3 的协同信号。
3. 如果 watcher 唤醒 Codex，Codex 仍必须先读 AGENTS、OpenSpec、Mattermost 最新消息和相关 PR 状态。
4. 自动化只能辅助提醒和审查，不替代 #1 最终判断。

## 首次 OpenSpec 开发测试

下面是一轮最小、可回滚、低风险的 OpenSpec 开发测试。它只改文档，不碰业务代码。

### 1. propose 阶段

先在 PowerShell 里用真实 CLI 创建 change：

```powershell
openspec new change <change-id>
openspec status --change <change-id>
```

```text
执行本轮 OpenSpec propose 阶段。
你可以处理 Git 和 OpenSpec，但必须遵守 AGENTS.md、openspec/AGENTS.md、openspec/specs/**、docs/governance/**。
先检查当前工作区、当前分支、远端状态、OpenSpec 状态和未忽略文件。
不要重复 openspec init。
如需运行 openspec update .，遇到删除 openspec/AGENTS.md 或 instruction 文件的提示，默认选择 n。
先输出你准备执行的 Git/OpenSpec 步骤；只有安全的只读检查可以直接执行。
涉及 commit、push、merge、rebase、删除、reset 的操作，必须说明原因和目标后再执行。
本轮 change-id: openspec-smoke-test。
本轮目标: 验证 OpenSpec 工作流，不改业务功能。
只做 propose：生成或检查 proposal.md、design.md、tasks.md、specs/governance/spec.md。
生成后停止，等待我审查，不要 apply。
```

### 2. Codex 应先确认 OpenSpec 状态

```powershell
openspec update .
openspec list
openspec validate --all
```

如果 `openspec update .` 要删除 `openspec/AGENTS.md`，选 `n`。

### 3. 创建测试 change

```powershell
New-Item -ItemType Directory -Force openspec/changes/openspec-smoke-test/specs/governance
```

创建这些文件：

```text
openspec/changes/openspec-smoke-test/proposal.md
openspec/changes/openspec-smoke-test/design.md
openspec/changes/openspec-smoke-test/tasks.md
openspec/changes/openspec-smoke-test/specs/governance/spec.md
```

### 4. 让 Codex 先读规范再动手

给 Codex 的输入：

```text
先读取 AGENTS.md、openspec/AGENTS.md、openspec/specs/**、openspec/changes/openspec-smoke-test/**、docs/governance/**。
本轮 change-id: openspec-smoke-test。
本轮只做 docs-only 的 OpenSpec 工作流测试，不改业务代码。
请先检查 proposal/design/tasks 是否完整，再给出要改的文档清单。
```

### 5. validate change

```powershell
openspec validate openspec-smoke-test --strict
openspec validate --all
```

### 6. 提交测试分支

```powershell
git add .
git commit -m "v1.2.6: test openspec collaboration workflow"
git push -u origin t1/v1.2/openspec-smoke-test
```

### 7. apply 与 archive

#1 审查通过后，才让 Codex 执行 apply。`apply` 不是 `openspec apply` 命令，而是让 Codex 按 `tasks.md` 实现代码。验证通过后，再执行真实 CLI：

```powershell
openspec archive <change-id> --yes
openspec validate --all
```

审查和 PR 按 [docs/governance/PR_REVIEW_RUNBOOK.md](docs/governance/PR_REVIEW_RUNBOOK.md) 执行。

## 2026-05-10 V1.6.5 RAG-Pro 本地模型与 Milvus Lite 阻断说明

### #2/#3 必看：离线模型包放置位置

云端 `main` 不会、也不应该包含 BGE 模型包。#2/#3 从云端 `main` 拉取代码后，必须把 #1 离线发送的两个模型包解压到下面两个固定目录：

```text
CarbonRag/data/outputs/models/BAAI/bge-m3
CarbonRag/data/outputs/models/BAAI/bge-reranker-v2-m3
```

以 Windows 默认项目路径为例：

```text
F:\Project\CarbonRag\data\outputs\models\BAAI\bge-m3
F:\Project\CarbonRag\data\outputs\models\BAAI\bge-reranker-v2-m3
```

解压后目录结构必须是：

```text
CarbonRag/
└─ data/
   └─ outputs/
      └─ models/
         └─ BAAI/
            ├─ bge-m3/
            │  ├─ config.json
            │  ├─ pytorch_model.bin
            │  ├─ tokenizer.json
            │  └─ ...
            └─ bge-reranker-v2-m3/
               ├─ config.json
               ├─ model.safetensors
               ├─ tokenizer.json
               └─ ...
```

不要放到：

- `backend/app/models`
- `backend/models`
- `C:\Users\<user>\.cache\huggingface`
- 任意 Git 会追踪的源码目录

不要多套一层目录，例如 `bge-m3\bge-m3\...` 是错误的。

放好后，Windows 开发者先启动 Docker Milvus，再运行本地 RAG smoke：

```powershell
powershell -NoProfile -ExecutionPolicy Bypass -File scripts/rag-start-milvus-docker-windows.ps1
powershell -NoProfile -ExecutionPolicy Bypass -File scripts/rag-smoke-milvus-standalone.ps1
powershell -NoProfile -ExecutionPolicy Bypass -File scripts/rag-smoke-local-model.ps1
```

至少应看到：

```text
bge_m3 2 1024 True
```

- #1 已把 RAG 默认 embedding provider 改回 `bge_m3`，普通聊天 API 不能替代 embedding API。后续 RAG-Pro 路线默认使用本地 `BAAI/bge-m3` 与 `BAAI/bge-reranker-v2-m3`。
- #1 已下载并 smoke 通过 BGE-M3 embedding：输出 `bge_m3 2 1024 True`，说明 dense 向量 1024 维且 sparse lexical weights 非空。
- 当前 Windows 本机 Milvus Lite 仍阻断：`pip install "milvus-lite>=2.5.1"` 返回 `No matching distribution found`。这不是漏装命令，而是当前 Windows/Python 环境拿不到可用发行包。
- RAG-Pro 的 Milvus Lite 方式是 `pymilvus.MilvusClient(uri="backend/data/milvus.db")` 本地文件 URI；在支持平台上会触发 Milvus Lite。V1.6.8 起，Windows 默认验收路线改为 Docker Milvus Standalone，WSL/Linux/macOS 才继续使用 Lite 文件模式。
- 禁止因为 Windows 本机 Milvus Lite 失败就恢复 hash embedding、假向量或“看起来成功”的 RAG。失败必须明确显示 degraded/failed。
- 可离线分发给 #2/#3 的大资产只有模型目录：`data/outputs/models/BAAI/bge-m3` 与 `data/outputs/models/BAAI/bge-reranker-v2-m3`，各约 2.14GB。不要打包 `backend/.conda`、`frontend/node_modules`、`.gitnexus`、`data/outputs/hf-cache`。
- #2/#3 从云端 `main` 拉取后，把 #1 离线发出的模型包解压到仓库内固定位置：`data/outputs/models/BAAI/bge-m3` 与 `data/outputs/models/BAAI/bge-reranker-v2-m3`。解压后不能多套一层目录，也不能放到 C 盘 Hugging Face 默认缓存或源码目录。
- `3rdparty/RAG-Pro` 原目录约 5.96GB，里面含本地环境/模型/索引内容，不建议整包发送。如需共享参考，只发裁剪源码包，排除 `backend/venv`、`backend/models`、`.gitnexus`、缓存和日志。
- 详细命令与打包建议见 [docs/PLAN/V1.6.5.md](docs/PLAN/V1.6.5.md)。
