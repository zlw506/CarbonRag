## 3. 项目当前定位

CarbonRag 的目标是构建一个面向中小企业的 **“双碳”政策与企业应用智能问答系统**。当前立项后的实施策略统一收缩为 **先做可跑通的云 API 版 MVP**，后续再逐步向本地开源模型与更强隐私架构演进。

因此，本项目当前不是“把所有双碳能力一次做完”，而是：

- 先完成核心链路跑通
- 先完成可展示的业务闭环
- 先为后续替换本地开源模型保留接口与结构
- 不在首轮为了“最终形态”而拖慢当前落地

---

## 4. 当前阶段统一目标

### 4.1 必做目标

1. **双碳政策知识库基础版**
   - 先覆盖国家级政策、北京市政策、部分关键标准与指南
   - 数据量先求“够演示、够问答、够验证”，不求一次性全覆盖
2. **RAG 问答主链路**
   - 用户输入问题
   - 系统检索相关政策或标准片段
   - 调用云端大模型 API 生成回答
   - 回答中尽量附带来源或引用依据
3. **企业私有资料接入基础版**
   - 支持导入少量企业样例数据或文档
   - 先做“公共知识 + 企业私有资料”联合检索
   - 先不做复杂企业级数据平台集成
4. **基础碳核算能力**
   - 至少支持一套简化碳核算流程
   - 能根据输入字段给出核算结果
   - 结果链路可演示、可解释
5. **报告生成基础版**
   - 能基于问答结果或核算结果生成报告草稿
   - 格式先以 Markdown、HTML 或 Word 文本输出为主
   - 不把排版系统做成独立大工程
6. **可演示的前后端闭环**
   - 至少有一个可操作界面或明确的 API 调用入口
   - 演示时可以完成“提问 -> 检索 -> 回答 -> 核算 -> 出报告”的关键路径

### 4.2 当前阶段不做或只留接口

- 本地开源大模型正式部署
- RTX4090 量化部署与推理优化
- 模型微调、LoRA、QLoRA
- 联邦学习、差分隐私、复杂脱敏系统
- 全量自动爬虫覆盖全国所有政策网站
- 多租户、复杂 RBAC 权限体系
- 生产级审计平台、告警平台、全套运维平台
- 高复杂度流程编排系统

## 5. 初期技术路线

### 5.1 当前原则

**初期实现统一使用云端 API。**

这意味着：

- 问答生成先接云端模型 API
- 报告润色先接云端模型 API
- 不在首轮把主要时间消耗在本地模型部署与算力调优上

### 5.2 重要约束

虽然当前统一使用云端 API，但项目结构必须保留未来替换为 **开源模型 / 私有部署模型** 的能力。也就是说：

- 模型调用必须经过统一封装层
- 不允许把具体供应商 API 写死在业务逻辑深处
- 提示词、模型名、温度、上下文长度等配置应外置
- 后续若切换到本地推理服务，应尽量不改上层业务逻辑

### 5.3 推荐的最简技术拆分

- **前端：** Web 页面或轻量管理页面
- **后端：** 提供 `/ask`、`/calc_carbon`、`/generate_report` 等接口
- **检索层：** 文档切分、Embedding、向量检索、RAG Prompt 拼接
- **模型层：** 统一封装云端 API 调用
- **数据层：** 公共知识库、企业样例私有库、排放因子与模板文件

### 5.4 当前推荐实现倾向

首轮应优先选择：

- 容易接入的后端框架
- 容易落地的向量方案
- 容易演示的前端页面
- 容易维护的文档结构

当前不鼓励在首轮引入过重、过杂、过难维护的中间件。

## 8. 项目技术边界与结构意识

### 8.1 公共知识部分

首轮重点来源建议包括：

- 国家发改委、生态环境部等国家级政策来源
- 北京市政府、首都之窗等地方政策来源
- 关键国家标准与核算指南
- 与中小企业申报、补贴、核算直接相关的高价值文件

当前落地以“高价值、少而精”为原则，不做全网铺开。

### 8.2 企业私有部分

首轮企业私有数据仅做样例级接入，典型形式包括：

- 电力、燃气等能耗数据表
- 生产量或原料使用表
- 项目申报材料或内部说明文档
- 交通、差旅、物流等示例记录

### 8.3 公私分离原则

必须明确：

- 公共政策知识与企业私有知识分开存放
- 仓库中原则上不直接提交真实敏感企业数据
- 私有样例若入库，必须是脱敏版或演示版
- 真正的企业原始数据不作为首轮 Git 管理对象

## 11. Git 纪律

### 11.1 核心目标

所有 Git 操作必须服务于四件事：

1. 能明确当前基线是什么
2. 能明确这轮改了什么
3. 改坏后能快速回退
4. 演示现象能和某个提交一一对应

### 11.2 施工前必须 checkpoint

**硬规则：**

- 未 checkpoint，不做高风险修改
- 未 checkpoint，不开始一轮正式施工
- 未 checkpoint，不允许“先试试再说”

**标准操作：**

```bash
git status -sb
git add .
git commit --allow-empty -m "checkpoint: before v0.x.x"
git push
```

### 11.3 结果完成后必须留结果提交

```bash
git status -sb
git add .
git commit -m "v0.x.x: <本轮内容摘要>"
git push
```

### 11.4 不在脏工作树上继续叠改

若当前工作树已经是脏的，必须先处理干净，再进入下一轮。

处理方式只有两种：

- 有价值：先提交 checkpoint
- 无价值：直接恢复

例如：

```bash
git restore .
```

### 11.5 Hotfix 只做最小修复

- hotfix 只允许最小范围改动
- hotfix 不允许顺手改架构
- hotfix 不允许把多个问题混修
- hotfix 的目标是恢复稳定，不是重构系统

### 11.6 回退优先 restore / revert

**未提交改坏：**

```bash
git restore .
```

**已提交且已 push 到共享分支：**

```bash
git revert HEAD
git push
```

**本地实验分支且确认无人依赖：**

```bash
git reset --hard HEAD~1
```

共享分支优先 `revert`，不要乱改历史。

## 12. 推荐分支模型

### 12.1 main

用途：

- 只放稳定、可演示、可交付内容
- 不直接承载高风险试错

### 12.2 dev

用途：

- 当前阶段主开发集成分支
- 通过联调后再择机合入 main

### 12.3 feature/*

用途：

- 新功能开发
- 模块新增
- 单点能力打通

建议命名：

- `feature/init-backend`
- `feature/rag-ingest`
- `feature/ask-api`
- `feature/carbon-calc`
- `feature/report-generate`

### 12.4 hotfix/*

用途：

- 从稳定点出发做最小修复

### 12.5 verify/*

用途：

- 纯验证历史版本
- 验证 last-known-good
- 演示前回归检查

## 13. 每轮施工标准流程

### 13.1 开工前

```bash
git status -sb
git checkout dev
git pull
git add .
git commit --allow-empty -m "checkpoint: before v0.x.x"
git push
git checkout -b feature/<topic>
git push -u origin feature/<topic>
```

### 13.2 施工中原则

- 一轮只解决一类问题
- 先打通主链路，再做美化与补细节
- 先能跑，再提精度
- 先保演示，再做扩展
- 不把多个目标混到同一轮

### 13.3 施工后

- 先自测
- 再联调
- 通过后提交结果
- 需要时打 tag
- 再合回 dev
